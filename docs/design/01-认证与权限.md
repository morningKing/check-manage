# 域① 认证与权限

> 架构图对应：业务域 ①（认证与权限）。索引见 [`README.md`](./README.md)，全局视图见 [`功能架构图.svg`](./功能架构图.svg)。

## 1. 模块职责

系统的横切安全层：签发与校验 JWT、提供数据驱动的自定义角色 RBAC，作为**唯一权威门禁**保护所有其它业务域。前端的权限判断仅用于 UX（按钮/菜单显隐），后端装饰器才是最终裁决。

## 2. 模块组成

| 层 | 组件 | 说明 |
|----|------|------|
| 蓝图 | `auth`（auth.py） | 登录、`/auth/me` 刷新当前用户与权限 |
| 蓝图 | `users` | 用户账号 CRUD |
| 蓝图 | `roles` | 自定义角色与权限矩阵 CRUD（`/admin/roles`） |
| 工具 | `server/auth.py` | JWT 装饰器：`login_required` / `write_required` / `require_permission` / `login_required_sse` / `api_key_required` |
| 工具 | `utils/permissions.py` | 权限解析与缓存，`PERMISSION_CATALOG` 能力目录 |
| 工具 | `utils/rbac_guard.py` | 数据页 CRUD 级 `require_page_action()` |
| 前端 | `stores/auth.ts` | token 管理、`isSuperuser`/`can()`/`canPage()`、路由守卫门禁 |
| 前端 | `stores/role.ts` · `views/admin/RoleManager.vue` | 角色管理页 |

## 3. 数据模型

- **`users`** — `id`、`username`、`password_hash`、`role`(角色 slug)、`display_name`。
- **角色与权限** — 内置 seed：**admin**（永久超级管理员，全权限，不可删）、**developer**（读写数据、无管理功能）、**guest**（只读）。`developer`/`guest` 可编辑，可新增自定义角色。权限矩阵数据驱动存储（见 `docs/custom-roles-rbac-guide.md`）。

权限三粒度：

| 粒度 | 载体 | 示例 |
|------|------|------|
| 管理功能开关 | `admin.*` 能力键 | `admin.users`、`admin.export_scripts`、`admin.ai_scan` |
| 数据页 CRUD | 角色 × pageId × 动作 | `canPage(pageId,'create'\|'update'\|'delete')` |
| 菜单可见性 | `menus.roles` JSONB | 角色 slug 在列表内才显示该菜单 |

## 4. 核心流程

**登录与鉴权链路**
1. `POST /auth/login` 校验密码 → `create_token()` 签发 JWT（payload 含 `userId/username/role/exp`，HS256）。
2. 前端存 `check-manage:token`（localStorage），Axios 拦截器注入 `Authorization: Bearer <token>`。
3. 每个受保护路由经装饰器：解析 Bearer → `decode_token()` → 失败返回 `未登录`/`登录已过期`(401)；按需再判角色或能力。
4. JWT 仅携带角色 slug，**权限编辑下次 `/auth/me` 即生效，无需重新登录**（权限在后端按 slug 实时解析 + 缓存）。

**装饰器语义**

| 装饰器 | 拦截 | 用途 |
|--------|------|------|
| `login_required` | 无有效 token → 401 | 仅需登录的读接口 |
| `write_required` | guest → 403 | 写操作（开发/管理员放行） |
| `require_permission('admin.x')` | 角色无该能力 → 403 | 管理功能 |
| `login_required_sse` | 同 login_required，额外接受 `?access_token=` | SSE/下载（浏览器 EventSource 无法设头） |
| `api_key_required` | 校验 API Key | Open API（见 [域⑥](./06-开放集成.md)） |

> ⚠️ **能力 ≠ 菜单可见性**：授予 `admin.x` 能力让角色能*访问*路由，但侧边栏链接只在该菜单 `roles` 含角色 slug 时才显示。

## 5. 关键接口

| 路由 | 方法 | 鉴权 | 作用 |
|------|------|------|------|
| `/auth/login` | POST | 公开 | 登录签发 JWT |
| `/auth/me` | GET | login_required | 刷新当前用户 + 权限集 |
| `/users` `/users/<id>` | CRUD | require_permission('admin.users') | 用户管理 |
| `/roles` `/roles/<id>` | CRUD | require_permission('admin.roles') | 角色与权限矩阵 |

## 6. 依赖与协作关系

- **被依赖**：几乎所有其它域都经本域装饰器鉴权；[域③](./03-动态数据核心.md) 写操作经 `write_required` + `require_page_action`。
- **依赖**：仅依赖 PostgreSQL（`users` 表）与配置中心（[域②](./02-配置中心.md) 的 `menus.roles` 决定菜单可见性）。
- **AI 链路特例**：[域⑨](./09-AI智能助手.md) 的 MCP Server 通过会话令牌反查用户角色，使 Agent 访问数据同样受本域 RBAC 约束。

## 7. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证方式 | 无状态 JWT | 水平扩展友好，无会话存储 |
| JWT 载荷 | 仅角色 slug | 权限改动免重登即时生效 |
| 权限模型 | 数据驱动可自定义角色 | 业务可增删角色与能力，不改代码 |
| 权威边界 | 后端装饰器为准 | 前端门禁仅 UX，杜绝绕过 |
| SSE 鉴权 | `?access_token=` 回退 | 浏览器 EventSource 无法设请求头 |
