# 域③ 动态数据核心

> 架构图对应：业务域 ③（动态数据核心）。索引见 [`README.md`](./README.md)。

## 1. 模块职责

平台的数据中枢。`dynamic` 蓝图是**万能 CRUD**（catch-all `/<collection>`），所有业务数据存于 `dynamic_data` 单表 JSONB。围绕它提供关联、关系图谱、查询、评论、时间线，并以**乐观锁**保障多用户并发一致性。

## 2. 模块组成

| 蓝图 | 职责 |
|------|------|
| `dynamic` | 通用 CRUD，分页/过滤/全文搜索，主键唯一校验，关联管理（catch-all，**必须最后注册**） |
| `relations` | M:N 关联读写（`data_relations` 表） |
| `relation_graph` | 关系图谱可视化数据 |
| `query` | MongoDB 风格查询控制台 |
| `comments` | 记录级评论 |
| `timeline` | 评论 + 操作日志合并时间线 |

前端：`stores/pageConfig.ts`（关联/引用/quote 解析、自增生成、批量操作、`addPageData`/`updatePageData`/`fetchPageData`）、`views/dynamic/DynamicPage.vue`（表格/看板/Excel/日历/甘特 多视图）、`components/dynamic-form/`（按 controlType 映射控件）。

## 3. 控件类型（controlType）

| 类型 | 说明 |
|------|------|
| text/textarea/richText/number | 输入类 |
| select/multiSelect/radio/checkbox | 选择类 |
| date/datetime · file/image | 日期 / 上传 |
| `relation` | M:N 双向关联（`data_relations`） |
| `reference` | 1:N 父子，字段继承 |
| `quoteSelect` | 单向多选引用 |
| `autoSequence` | 自增 ID（如 IC-001） |
| `autoTimestamp` | 创建/更新自动填充 |
| `compositeText` | 只读，按 `sourceFields` 拼接 |

字段可携带 `workflowConfig` 驱动状态机（见 [域②](./02-配置中心.md)）。

## 4. 数据模型

**`dynamic_data`**（核心表，单表 JSONB）

| 列 | 类型 | 说明 |
|----|------|------|
| id | VARCHAR(100) PK | 记录唯一标识（前端生成 `<collection>-<uuid8>`） |
| collection | VARCHAR(200) | 由 pageId 推导，区分实体 |
| data | JSONB | 全部业务字段 |
| created_at / updated_at | TIMESTAMPTZ | 时间戳 |
| **version** | INTEGER | **乐观锁版本号**，新建为 1，每次修改 +1 |
| branch_id | VARCHAR | 版本分支隔离（见 [域⑦](./07-版本协作.md)） |

**`data_relations`** — 存 M:N 关系，按 `(collection, record_id, field_name, branch_id)` 查询。
**`record_comments`** — `id`、`collection`、`record_id`、`content`、`mentions` JSONB、`author_*`、`created_at`。

## 5. 核心流程

### 5.1 CRUD 与字段自动化
- **创建**：前端 `addPageData` 生成 id、填充 `autoTimestamp`/`autoSequence`/`compositeText`、合并关联数据 → `POST /<collection>`；后端校验主键唯一、跑校验脚本（[域④](./04-脚本扩展.md)）后插入，`version=1`。
- **读取**：`GET /<collection>` 支持 `page/pageSize/all=true`、`q`（Mongo 查询）、`keyword`（全文）、`locateId`；`row_to_record` 将 `version` 映射为 `_version` 返回。
- **行内复制新增**：操作栏「复制」按钮预填充整行（剔除 `id` 与 autoSequence/autoTimestamp 字段），以新增模式打开表单供人工检视后入库。

### 5.2 并发控制 — 乐观锁（核心设计）

选型：内部管理系统冲突概率低、对一致性要求高 → **乐观锁**（提交时校验版本，冲突拒绝），避免悲观锁的锁等待与死锁。

**更新双重校验**（防丢失更新 + 防竞态）：

```
PUT 请求(_version=N)
  → ① 读 db_version
  → ② 前置比较: client_version != db_version → 409 VERSION_CONFLICT（快速失败）
  → ③ UPDATE ... SET version=N+1 WHERE id=? AND version=N
  → ④ rowcount==0 → 409（捕获"前置通过后、UPDATE 前"的极端竞态）
  → 返回 200 {_version: N+1}
```

| 校验层 | 防御场景 |
|--------|----------|
| ② 前置比较 | 客户端版本明显过期 |
| ③ WHERE version=N | SELECT 与 UPDATE 间的竞态窗口（数据库级原子约束） |

**前端配合**：版本号经 `pageDataCache` 传递（GET 取 → PUT 带 → 响应更新）；收到 `VERSION_CONFLICT` 时关闭编辑框并自动 `loadPageData()` 刷新；Axios 拦截器对 `code===VERSION_CONFLICT` 的 409 跳过自动提示，避免与业务层重复弹窗。

**边界**：ETL/扫描等服务端批量写入跳过客户端校验但仍 `version+1`；关联同步在单事务内完成、不涉及 version（靠事务原子性）；删除不校验版本（有引用依赖保护）；不带 `_version` 的旧客户端跳过前置检查仍受 ③ 保护（向后兼容）。

### 5.3 关联同步与时间线
- 关联写入在单事务内完成正向 + 反向（`ON CONFLICT DO NOTHING`），要么全成功要么全回滚。
- 时间线：运行时 `UNION` 合并 `record_comments`（type=comment）与 `operation_logs`（type=change/statusChange），按 timestamp 排序返回，避免冗余存储。

## 6. 关键接口

| 路由 | 方法 | 说明 |
|------|------|------|
| `/<collection>` | GET/POST | 列表(分页/过滤/搜索) / 创建 |
| `/<collection>/<id>` | GET/PUT/DELETE | 详情 / 乐观锁更新 / 删除 |
| `/relations/<coll>/<id>/<field>` | GET/PUT | 关联读写 |
| `/relation-graph/...` | GET | 关系图谱 |
| `/query` | POST | Mongo 风格查询 |
| `/comments/<coll>/<rid>` | GET/POST | 评论 |
| `/timeline/<coll>/<rid>` | GET | 合并时间线 |

**保留集合路径**（不可作为业务 collection）：`menus`/`pageConfigs`/`relations`/`auth`/`users`/`ai`/`webhook` 等，权威列表见 `routes/dynamic.py:RESERVED`。

## 8. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 数据存储 | 单表 `dynamic_data` JSONB | 新增实体/字段零迁移 |
| 并发控制 | 乐观锁（version 列） | 冲突少、无锁等待；单调整数比时间戳精确 |
| 冲突防御 | 前置比较 + WHERE 双重 | 单层无法覆盖竞态窗口 |
| 冲突响应 | 409 + `VERSION_CONFLICT` + 自动刷新 | HTTP 语义准确、体验优于手动刷新 |
| 关联存储 | 独立 `data_relations` + 事务 | 不参与版本号，靠事务原子性 |
| 时间线 | 运行时 UNION | 评论与日志各自独立，免冗余 |
