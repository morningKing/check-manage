## 3. 看板视图

### 3.1 数据流

```mermaid
sequenceDiagram
    participant User as 用户
    participant DP as DynamicPage
    participant KB as KanbanBoard
    participant Store as PageConfigStore
    participant API as PUT /collection/:id

    User->>DP: 切换到看板视图
    DP->>KB: 传入 data, groupField, groupOptions
    KB->>KB: 按 groupField 分列渲染卡片

    User->>KB: 拖拽卡片到新列
    KB->>DP: emit card-move(recordId, newValue)
    DP->>Store: updatePageData(pageId, recordId, data)
    Store->>API: PUT {field: newValue, _version}
    API-->>Store: 200 OK / 400 工作流拒绝
    alt 成功
        Store-->>DP: 更新本地数据
        DP-->>KB: 卡片保持在新列
    else 失败
        Store-->>DP: 抛出异常
        DP->>DP: loadPageData() 重新加载
        DP-->>KB: 卡片回弹到原列
    end
```

### 3.2 view_config 数据结构

```json
{
  "defaultView": "table | kanban",
  "kanban": {
    "groupField": "status",
    "cardTitle": "taskName",
    "cardFields": ["assignee", "priority", "dueDate"],
    "columnOrder": ["todo", "inProgress", "done"],
    "cardColorField": "priority"
  }
}
```

### 3.3 组件设计

```mermaid
graph LR
    subgraph KanbanBoard
        A[props.data] --> B[columnDataMap]
        B --> C1[Column: todo]
        B --> C2[Column: inProgress]
        B --> C3[Column: done]
        C1 --> D1[vuedraggable]
        C2 --> D2[vuedraggable]
        C3 --> D3[vuedraggable]
        D1 & D2 & D3 --> E{onDragChange}
        E -->|evt.added| F[emit card-move]

    G[props.searchKeyword] --> B
```

- 使用 `vuedraggable` 的 `group="kanban"` 实现跨列拖拽
- 每列维护独立的可变数组 `columnDataMap`，在 `data` 或 `searchKeyword` 变更时重建
- 卡片颜色映射：`high/urgent/紧急` → 红色, `medium/中` → 橙色, `low/低` → 绿色

## 4. 状态工作流引擎

### 4.1 校验流程

```mermaid
flowchart TD
    A[PUT 更新请求] --> B{字段有 workflowConfig?}
    B -->|否| Z[正常更新]
    B -->|是| C{状态值发生变更?}
    C -->|否| Z
    C -->|是| D[find_transition<br/>from → to]
    D --> E{找到转换规则?}
    E -->|否| F[返回 400<br/>不允许的状态转换]
    E -->|是| G{检查角色权限}
    G -->|无权限| H[返回 400<br/>角色无权执行]
    G -->|通过| I{检查前置条件}
    I -->|不满足| J[返回 400<br/>条件不满足 + message]
    I -->|通过| K[execute_actions]
    K --> K1[setField: 写入字段值]
    K --> K2[runScript: 执行脚本]
    K1 & K2 --> Z
```

### 4.2 workflowConfig 数据结构

```json
{
  "enabled": true,
  "transitions": [
    {
      "from": "todo",
      "to": "inProgress",
      "label": "开始处理",
      "roles": ["admin", "developer"],
      "conditions": [
        { "field": "assignee", "rule": "notEmpty", "message": "请先指派负责人" }
      ],
      "actions": [
        { "type": "setField", "field": "startTime", "value": "$NOW" }
      ]
    }
  ]
}
```

### 4.3 多端一致性校验

```mermaid
graph TB
    subgraph 前端预校验
        A1[SelectInput 禁用不可达选项]
        A2[WorkflowActions 检查 conditions]
        A3[KanbanBoard 拖拽失败回弹]
    end

    subgraph 后端强校验
        B1[dynamic.py PUT handler]
        B2[workflow.validate_transition]
        B3[返回 400 错误]
    end

    A1 & A2 & A3 -->|HTTP PUT| B1
    B1 --> B2
    B2 -->|不允许| B3
```

前端校验提升用户体验（禁用不合法选项、条件提示），后端校验保证数据一致性。

- 通知记录包含 `source_collection` 和 `source_record_id`，支持精准定位

## 8. 跨集合联动规则

### 8.1 触发执行流程

```mermaid
sequenceDiagram
    participant Client as 前端
    participant API as dynamic.py
    participant Engine as trigger_engine
    participant DB as 数据库
    participant Log as trigger_logs

    Client->>API: POST/PUT/DELETE 操作
    API->>API: 执行主操作（CRUD）
    API->>Engine: fire_triggers(event, collection, ...)
    Engine->>DB: SELECT FROM trigger_rules<br/>WHERE source_collection AND enabled

    loop 每条匹配规则
        Engine->>Engine: 检查 trigger_condition
        alt 条件满足
            Engine->>Engine: _execute_action()
            alt action_type = create
                Engine->>DB: INSERT INTO dynamic_data (target_collection)
            else action_type = update
                Engine->>DB: UPDATE dynamic_data WHERE matchField
            else action_type = runScript
                Engine->>Engine: script_runner.run()
            end
            Engine->>Log: _log_trigger(status=success)
        else 条件不满足
            Note over Engine: 跳过该规则
        end
    end
```

### 8.2 表达式解析

```mermaid
graph LR
    A[字段映射表达式] --> B{"$source." 前缀?}
    B -->|是| C[从源记录取字段值<br/>source_data.get field]
    B -->|否| D{"$operator"?}
    D -->|是| E[操作人用户名]
    D -->|否| F{"$NOW"?}
    F -->|是| G[当前 ISO 时间]
    F -->|否| H[作为字面量值]
```

## 11. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 工作流配置存储位置 | `FieldConfig.workflowConfig` | 跟随字段定义，不新增表，配置即代码 |
| 看板配置存储位置 | `page_configs.view_config` | 页面级配置，不影响字段结构 |
| 联动规则独立表 | `trigger_rules` | 规则可跨集合，不适合嵌入单个 PageConfig |
| 通知不阻塞主操作 | `try/except` 包裹 | 通知是辅助功能，不能影响核心 CRUD 可靠性 |
| 前后端双重校验 | 前端禁用 + 后端拒绝 | 用户体验 + 数据一致性并重 |
| 时间线合并查询 | 运行时 UNION 排序 | 避免冗余存储，评论和日志各自独立维护 |
| 仪表盘聚合 | PostgreSQL JSONB 查询 | 复用现有 dynamic_data 单表，无需额外 ETL |
| 乐观锁 + 工作流 | 先版本检查再工作流校验 | 防止并发冲突导致的非法状态转换 |
