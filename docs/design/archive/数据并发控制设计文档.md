## 历史背景（保留备查）

本文档最初记录的核心问题是多用户并发编辑同一条（含丰富关联关系的）记录时的**丢失更新**：

```
用户A 读取记录 → 用户B 读取同一记录 → 用户B 保存 → 用户A 保存（覆盖了 B 的修改）
```

### 方案选型

在悲观锁、乐观锁、最后写入胜出三者中选择**乐观锁**，理由：内部管理系统同时编辑同一记录概率低、
对一致性有强要求、无需引入分布式锁基础设施。

### 核心机制（已并入并扩展进 `../10-数据并发控制.md` §3）

给 `dynamic_data` 增加 `version` 列做乐观锁：

- **读取**：`row_to_record` 把 `version` 映射为 `_version` 返回；前端缓存于 `pageDataCache`。
- **更新**：双重校验 —— ① 前置比较 `client_version != db_version` 快速失败；
  ② 带版本条件的 `UPDATE … SET version=N+1 WHERE id=… AND version=N`，按 `rowcount=0` 捕获
  「校验通过后、UPDATE 之前」的极端竞态。冲突均返回 `409 VERSION_CONFLICT`。
- **字段级 merge**（`{**old, **new}`）：改动不同字段的并发互不覆盖。
- **关联同步**：`data_relations` 的双向同步在单事务内完成，不涉及版本号，靠事务原子性保证。
- **前端**：`409 VERSION_CONFLICT` 由 `conflict.ts` 统一处理（提示 + 自动刷新 + 基于最新值重做），
  响应拦截器对该 code 跳过自动弹窗以免重复提示。
- **向后兼容**：不携带 `_version` 的旧客户端跳过前置比较，仍受数据库级 `WHERE version=N` 保护。
