# 巡检用例管理系统 — 功能架构说明

> 配套架构图：[`功能架构图.svg`](./功能架构图.svg)　·　文档索引：[`README.md`](./README.md)
>
> 本文是**跨域总览**。每个业务域的模块组成、数据模型、核心流程、接口与设计决策详见其专属文档（下文各域标题均带链接）。

本文档逐层解释系统的功能模块、模块间的数据流与依赖关系，以及与外部系统的集成点。系统采用**配置驱动**架构：不为每个业务实体编写页面或建表，而是通过 `PageConfig`（字段 schema）+ `Menu`（菜单）配置，由系统自动生成 UI 与 API。

---

## 一、整体分层

系统自上而下分为 5 个逻辑层 + 1 个独立 AI 子系统：

| 层 | 职责 | 进程/端口 |
|----|------|-----------|
| 外部参与者 & 集成点 | 浏览器用户、Open API 调用方、第三方 HTTP 端点、LLM Provider | — |
| 前端 SPA | Vue 3 单页应用，视图/状态/请求/动态路由 | Vite :5173 |
| Flask 业务域 | 35 个蓝图按 9 个业务域分组的 API 网关 | Flask :3002 |
| 引擎与后台调度 | 进程内常驻的脚本沙箱、ETL/Webhook/批任务/扫描引擎 + 3 个调度器 | Flask 进程内 |
| 数据与存储 | PostgreSQL 单表 JSONB + 仓库外文件存储 | PostgreSQL :5432 |
| AI 子系统（独立） | OpenCode 运行时 + MCP Server | OpenCode :4096 / MCP :3003 |

**核心数据流主线**：浏览器 → Vue SPA →（`/api` 代理，携带 JWT）→ Flask 业务域 → 引擎 → PostgreSQL；AI 链路则是 Flask → OpenCode →（MCP 工具）→ PostgreSQL。

## 二、前端 SPA（Vue 3）

单页应用，由 4 个子模块组成：

- **视图层 Views** — `DynamicPage.vue` 是唯一的业务数据页面，根据 `pageId` 读取配置渲染表格/看板/Excel/日历/甘特视图；统一的**设置中心**（master-detail 收纳数据工具与系统配置，含工作流设计器 `WorkflowManager`）、工作流**收件箱** `WorkflowInbox`、`home` 首页、`ai-chat` 助手抽屉。
- **状态层 Pinia Stores** — `auth`（JWT/权限）、`menu`（菜单树缓存）、`pageConfig`（中央 store，含关联解析/自增生成/批量操作）、`systemConfig`（首页区块）、`aiChat`/`aiChatBatches`/`aiScanTask` 等。
- **API 层（Axios）** — 统一基址 `/api`，请求拦截注入 `Bearer <token>`，响应统一错误处理与 401 跳转。
- **动态路由 Router** — `dynamicRoutes.ts` 运行时读取 `menus` 表生成路由，统一指向 `DynamicPage.vue`；路由守卫做 RBAC 前端门禁（仅 UX，后端为权威）。

**依赖**：前端只与 Flask 通信，所有能力经 `/api` 获取；无直连数据库或 OpenCode。

## 三、Flask 业务域（9 组，35 蓝图）

> 蓝图注册顺序重要：`dynamic_bp`（catch-all `/<collection>`）必须最后注册以免遮蔽具体路由；`ai_chat_bp` 紧邻其前。

### ① 认证与权限（auth · users · roles）　→ [详细设计](./01-认证与权限.md)
JWT 签发与校验，数据驱动的自定义角色 RBAC。三粒度权限：管理功能开关（`admin.*`）、数据页 CRUD、菜单可见性。装饰器 `@require_permission` / `@write_required` 为权威门禁。
**被依赖**：几乎所有其它域都经其装饰器鉴权。

### ② 配置中心（menus · page_configs · column_views · system_config · home_widgets）　→ [详细设计](./02-配置中心.md)
系统的"元数据"层：菜单树、页面字段 schema、列视图、系统配置、首页区块（含**快速录入表单**区块——点击弹出表单直接入库关联数据页）。
**关系**：动态数据核心（③）与前端动态路由都**读取**本域的配置来决定 UI 与 API 行为。

### ③ 动态数据核心（dynamic · relations · relation_graph · query · comments · timeline · workflows）　→ [详细设计](./03-动态数据核心.md)
平台的数据中枢。`dynamic` 是万能 CRUD（catch-all），所有业务数据存于 `dynamic_data` 表 JSONB 列，按 `collection` 区分，按 `branch_id` 隔离分支。关联（M:N relation / 1:N reference / quoteSelect）、关系图谱、MongoDB 风格查询、评论、时间线。写入受**并发一致性**保护（乐观锁 + 序号原子分配 + 手填主键 advisory 锁 + 全写入路径序号不变式，详见横切文档 [⑩ 数据并发控制](./10-数据并发控制.md)）。
**跨页工作流引擎**（`workflows` 蓝图：`workflow_definitions` / `workflow_instances` / 收件箱 / 设计器）挂在 `update_item` 状态字段转换之后：按阶段定义**原子分配并生成下游记录**、推进/回退实例、按角色派发待办通知；下游 spawn 与回退重置严格落在源记录**同一分支**，推进失败即回滚整笔事务。
**依赖**：读 ②的字段配置；写时触发 ④校验脚本、⑥联动/Webhook、工作流编排，并向 ⑧ 发送待办通知。

### ④ 脚本扩展（export_scripts · validation_scripts）　→ [详细设计](./04-脚本扩展.md)
两类用户自定义 Python 脚本。导出脚本（页面级/菜单级，输出 json/csv/xml/txt/html）；校验脚本绑定页面，在新增/编辑写库前执行。二者均经 **script_runner 沙箱**执行（超时 + 禁 `import`/`eval`/`open` 等）。
**关系**：③在写库前调用校验脚本；导出功能调用导出脚本。

### ⑤ 数据流转（etl_tasks · menu_export · data_files）　→ [详细设计](./05-数据流转.md)
ETL 任务（抽取/转换/字段映射/条件过滤/写入集合，经 **etl_engine**）、菜单级批量导出 ZIP、文件/图片字段存储。
**集成点**：ETL 可从**第三方 HTTP 源**抽取数据。

### ⑥ 开放集成（api_keys · open_api · webhook · trigger_rules）　→ [详细设计](./06-开放集成.md)
对外暴露与事件驱动。`open_api` 让**外部系统**以 API Key 认证按集合读写数据；`webhook` 在 create/update/delete/merge 事件触发并向**第三方端点** POST（HMAC-SHA256 签名 + 重试，经 **webhook_engine**）；`trigger_rules` 数据联动（经 **trigger_engine**）。

### ⑦ 版本协作（project_versions · cross_project_deps）　→ [详细设计](./07-版本协作.md)
项目级分支/快照（创建/合并/锁定，跨分支数据隔离）；跨项目依赖三类型（track-main / read-write / read-only），依赖校验 + 通知。
**关系**：分支隔离作用于③的 `branch_id`；依赖变更经 **dependency_scheduler** 周期校验，结果写⑧通知。

### ⑧ 运维监控（operation_logs · backups · notifications · dashboards）　→ [详细设计](./08-运维监控.md)
操作审计日志、系统备份（手动/定时，经 **backup_scheduler**；支持还原与差异对比）、用户通知、仪表盘图表。
**关系**：接收⑦的依赖通知；备份覆盖全库。

### ⑨ AI 智能助手（ai · ai_chat · ai_chat_batches · ai_chat_prompt_templates · ai_scan_tasks）　→ [详细设计](./09-AI智能助手.md)
- **ai** — AI 全局配置。
- **ai_chat** — 对话会话与 SSE 流式代理；浏览器只连 Flask，Flask 作为网关代理 OpenCode 的事件流，并在 `message.finished` 持久化消息。支持 @子智能体、模型选择。
- **ai_chat_batches** — 批任务：选 N 文件 + 1 Prompt → N 个隔离子会话并发处理（**batch_engine + BatchWorker**，线程池限流 3）。
- **ai_chat_prompt_templates** — Prompt 模板 CRUD。
- **ai_scan_tasks** — 定时 AI 流水线：按计划扫描数据页，把待处理记录交给 AI 处理，解析结构化 JSON 回写记录（**ai_scan_engine + ai_scan_scheduler**），状态流转 待处理→处理中→已处理/处理失败。
**集成点**：本域是与 AI 子系统的唯一桥梁。

## 四、引擎与后台调度（Flask 进程内）

均在 `app.py` 启动，受 `WERKZEUG_RUN_MAIN` 守卫避免 Flask 重载双启：

| 组件 | 职责 | 被谁触发 |
|------|------|----------|
| **script_runner 沙箱** | 安全执行导出/校验脚本 | ④脚本扩展、⑤导出 |
| **etl_engine** | ETL 步骤编排执行 | ⑤ ETL 任务 |
| **webhook_engine / trigger_engine** | 事件触发 + 重试 / 数据联动 | ③数据写入事件 |
| **workflow_engine**（同步钩子，非调度器） | 状态转换→阶段推进/回退 + 原子 spawn 下游记录，与状态写入同事务 | ③ `update_item` 状态字段转换 |
| **batch_engine + BatchWorker** | 守护线程 + 线程池(3)，FIFO 认领子会话 | ⑨批任务、定时扫描 |
| **ai_scan_engine** | 原子认领记录→处理中，`jsonb_set` 回写 | ai_scan_scheduler |
| **调度器 ×3** | backup / dependency / ai_scan（APScheduler） | 定时 |

> 定时 AI 流水线复用批任务引擎：每次扫描 = 一个 batch，每条记录 = 一个子会话；批任务完成钩子回调 `ai_scan_engine.on_child_finished` 解析并回写。

## 五、数据与存储

### PostgreSQL（库 `casemanage`，单表 JSONB）
所有业务数据存于 `dynamic_data` 的 `data` JSONB 列，按 `collection` 区分实体、`branch_id` 隔离分支，**新增实体/字段无需迁移**。连接池 `SimpleConnectionPool(1-10)`。主要表分组：

- **核心**：dynamic_data · menus · page_configs · data_relations · users · dynamic_sequences（序号原子分配计数表）
- **工作流**：workflow_definitions · workflow_instances（跨页阶段编排）
- **版本**：project_versions · project_dependencies · project_dependency_relations
- **集成**：webhook_rules / webhook_logs · api_keys · trigger_rules
- **脚本**：export_scripts · validation_scripts · etl_tasks / etl_logs
- **运维**：operation_logs · backups / backup_settings · notifications
- **AI**：ai_chat_sessions / messages · ai_chat_batches · ai_chat_prompt_templates · ai_scan_tasks · ai_settings
- **其它**：dashboards · home_widgets · system_config · column_views · record_comments

### 文件存储（仓库外，避免被 AI 文件工具误读 & 不入 git）
- `DATA_FILES_ROOT` — 数据页文件/图片字段
- `AI_WORKSPACE_ROOT` — 每个 AI 会话的隔离工作区
- `batch-staging` — 批任务上传暂存
- `backups` — 系统备份文件

## 六、AI 子系统（独立进程，关键集成点）

这是系统唯一的**多进程**部分，与 Flask 解耦：

### OpenCode 运行时（`opencode serve` HTTP :4096）
Agent 编排、工具调用、SSE 事件流，支持 primary/subagent 多智能体。实际推理调用外部 **LLM Provider**（Claude / MiMo 等）。

### MCP Server（FastAPI + MCP Streamable-HTTP :3003）
向 Agent 暴露平台能力（数据查询/读取）。其设计要点：
- 与 Flask **解耦**，独立 venv 运行。
- 通过 opaque token（注册时由 Flask 生成）经 **DB 校验**，推导出用户身份与 RBAC，因此 Agent 访问数据**受同样的权限约束**。
- 直连 PostgreSQL 执行查询。

### AI 调用链路（图中橙色流）
1. Flask `ai_chat` 创建会话工作区 + token，请求 OpenCode 启动绑定该工作区的会话，并注册 MCP url `http://<mcp>/mcp?token=<token>`。
2. 浏览器经 Flask SSE 代理订阅 OpenCode 事件流。
3. OpenCode 推理时调用 LLM Provider；需要平台数据时经 MCP Server（带 token）→ MCP 校验后查 DB。

**设计意图**：给 OpenCode 增加一个 skill 即可让前端获得新能力，零前端改动。

## 七、关键依赖关系总览

```
配置中心② ──(读 schema)──▶ 动态数据核心③ ──(写前)──▶ 脚本④校验 / 集成⑥联动·Webhook
   ▲                              │
   │(读路由)                       ├──(update_item 状态转换)──▶ 跨页工作流引擎（阶段推进/回退·角色待办）
前端动态路由                       │
                                  │(分支隔离 branch_id)
                                版本协作⑦ ──(依赖变更)──▶ 运维⑧通知
                                   │
认证①  ──(装饰器鉴权)──▶  所有业务域
                                   │
AI⑨ ──(SSE 代理)──▶ OpenCode ──(MCP+token)──▶ MCP Server ──(RBAC 查询)──▶ PostgreSQL
                        │
                        └──(推理)──▶ LLM Provider
```

- **配置 → 数据**：②的字段 schema 决定③如何渲染与校验，是最核心的"元数据→数据"依赖。
- **数据 → 扩展/集成**：③的写操作是④校验、⑥Webhook/联动、⑨定时扫描的事件源。
- **数据 → 工作流**：③的 `update_item` 状态字段转换触发 **workflow_engine**（同步钩子，与状态写入同事务），原子生成下游阶段记录、推进/回退实例并派发角色待办（详见 [工作流引擎指南](../workflow-engine-guide.md)）。
- **认证横切**：①被所有域依赖。
- **AI 单向桥**：⑨是平台与 AI 子系统的唯一连接点；MCP 反向回连 DB 但受 RBAC 约束。

## 八、外部集成点小结

| 外部系统 | 接入方式 | 对应模块 |
|----------|----------|----------|
| 浏览器用户 | HTTPS + JWT，经 `/api` 代理 | 前端 SPA ↔ 全部业务域 |
| 外部系统（Open API 调用方） | REST + API Key | ⑥ open_api / api_keys |
| 第三方 HTTP 端点（Webhook 接收端） | 出站 POST + HMAC-SHA256 | ⑥ webhook + webhook_engine |
| 第三方 HTTP 源（ETL） | 出站 HTTP 抽取 | ⑤ etl_tasks + etl_engine |
| OpenCode 运行时 | Flask SSE 代理 + HTTP API | ⑨ ai_chat |
| MCP Server | OpenCode 经 MCP 协议 + token | MCP Server ↔ PostgreSQL |
| LLM Provider | OpenCode 内部调用 | OpenCode |
