# 域⑨ AI 智能助手

> 架构图对应：业务域 ⑨（AI 智能助手）+ 右侧 AI 子系统（OpenCode / MCP Server）。索引见 [`README.md`](./README.md)。

## 1. 模块职责

平台与 AI 能力的唯一桥梁。提供三种形态：**对话助手**（流式、工具调用、制品）、**批任务**（N 文件并发处理）、**定时 AI 流水线**（扫描数据页→AI 处理→结构化回写）。核心理念：**浏览器只与 Flask 通信**；Flask 作为网关与 SSE 代理对接 **OpenCode** Agent 运行时；平台能力经**独立 MCP Server** 暴露给 Agent，且受同一套 RBAC 约束。"加 skill 即获得能力，前端零修改"。

## 3. 外部子系统（独立进程）

| 进程 | 地址 | 职责 |
|------|------|------|
| OpenCode | `opencode serve` :4096 | Agent 编排、工具调用、SSE 事件流、调用 LLM Provider |
| MCP Server | :3003 | 向 Agent 暴露平台工具，token→DB 校验得用户/RBAC，直连 DB |
| LLM Provider | 外部 | Claude / MiMo 等大模型，由 OpenCode 内部调用 |

> **Agent 与模型可选**：对话/批任务/定时任务均可指定 OpenCode Agent（如 `build`/`plan`/`general`/`explore` 及子智能体），留空用默认；模型经 `OPENCODE_MODEL`（默认空 → OpenCode 选首个 provider 默认模型）。Agent 列表来自 `/ai/chat/agents`（需 OpenCode 在线）。

## 4. 数据模型

- **`ai_chat_sessions`** — `id`、`user_id`、`title`、`opencode_session_id`、`workspace_path`、`session_token`(唯一 opaque)、`token_expires_at`、`status`；批任务子会话另有 `batch_id`/`batch_seq`/`batch_input_file`；扫描子会话另有 `scan_task_id`/`source_record_id`。
- **`ai_chat_messages`** — `session_id`、`role`、`content` JSONB(typed parts：text/file/tool_use/run_result)。
- **`ai_chat_batches`** — `name`、`prompt`、`template_id`、`total/done/failed`、`status`、**`agent`**。
- **`ai_chat_prompt_templates`** — 按用户的 Prompt 模板。
- **`ai_scan_tasks`** — 扫描配置（collection、状态字段与各状态值、过滤、提示词模板、字段映射、调度间隔、每次条数、**agent**）。

## 5. 核心流程

### 5.1 对话（SSE 流式）
1. **建会话**：建仓库外 workspace → 落库 → 生成 opaque 令牌 → 写 `opencode.json`（含 MCP url `?token=` 与模型）→ `POST /session?directory=` → 回填 OpenCode 会话 id。
2. **发消息**：落库 user 消息（附件文本内联）→ `send_prompt_async(parts, model, agent, agent_parts)`。
3. **流式**：浏览器 `EventSource` 订阅 Flask SSE 代理（带 `?access_token=`，OpenCode 全局 `/event` 按 `sessionID` 过滤）→ store 按 part id 覆盖累积（text/reasoning/tool）→ `session.idle` 落库 assistant 文本、刷新 outputs。
4. **身份链路**：Flask 令牌 → workspace `opencode.json` → OpenCode 带令牌连 MCP → MCP 查表得 `user_id/role` → 工具 RBAC。
5. **制品**：助手大代码/文档块前端提取为制品卡（同名跨消息归为多版本）；HTML/SVG 沙箱 iframe 渲染；Python 制品可手动「运行」，结果作为 `run_result` 持久化。

### 5.2 批任务
1. 选 N 文件 + 1 Prompt（可选模板/Agent）→ 创建一个 `ai_chat_batches` + N 个 `pending` 子会话。
2. **BatchWorker**（守护线程 + `ThreadPoolExecutor(3)`）FIFO `FOR UPDATE SKIP LOCKED` 认领子会话，逐个 `create_session → send_message(agent=) → 等待完成 → 落库`，30 分钟/子会话超时。
3. 失败子会话红旗标记、不中止整批；`retry-failed` 把失败重置为 `pending`。Flask 重启时审计把孤儿 `running` 重置 `pending`。
4. 模板下拉每次展开/打开懒加载，确保"管理模板"新建后即可选到。

### 5.3 定时 AI 流水线（复用批引擎）
1. `ai_scan_scheduler`（APScheduler 1 分钟 tick，per-task 到期/锁、启动孤儿清扫）触发到期任务。
2. `ai_scan_engine` 原子 `FOR UPDATE SKIP LOCKED` 认领待处理记录 → 翻转"处理中"；上下文目录 = `record.md` + `attachments/`；prompt = 模板 + 按字段映射自动追加的 JSON 契约。
3. 每次扫描 = 一个 batch，每条记录 = 一个子会话（带 `scan_task_id`/`source_record_id`/`agent`）；批任务完成钩子 `batch_engine._run_one` 回调 `ai_scan_engine.on_child_finished` 解析 AI 的结构化 JSON，经参数化 `jsonb_set` 回写记录，状态流转 待处理→处理中→已处理/处理失败。
- 跨表副作用交给 skill 经 MCP；仅同行结构化回写内建。

## 7. 依赖与协作关系

- **依赖**：[域①](./01-认证与权限.md)（会话 `write_required`、MCP 经令牌反查 RBAC）、[域③](./03-动态数据核心.md)（扫描读写记录）、[域②](./02-配置中心.md)（字段配置）。
- **外部集成**：OpenCode（SSE 代理）→ MCP Server（工具，回连 DB 受 RBAC）→ LLM Provider（推理）。
- **复用**：定时流水线复用批任务引擎；前端动态路由对 `/ai-chat` 放行非访客（与后端 `@write_required` 对齐）。

## 8. 安全与隔离

- **令牌**：opaque、每会话唯一、可吊销、有 TTL；删会话先吊销令牌+标记删除，再尽力清目录。
- **工作区隔离**：`AI_WORKSPACE_ROOT/<user>/<session>/` 在**仓库外**——OpenCode 内置文件工具 cwd 是仓库根，故扫不到任何会话上传；访问上传只能经 `read_upload`（会话作用域）或服务端内联（文本、200KB 上限）。
- **路径穿越防护**：`safe_resolve`/`secure_filename`/`commonpath`。
- **密钥**：LLM provider 密钥只存 OpenCode 全局配置，不进 git。

## 9. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 接入方式 | 对接 OpenCode 而非直连 LLM | 获得 Agent 编排/工具/skill 生态，前端零改动加能力 |
| 浏览器边界 | 只连 Flask | 不暴露 OpenCode/MCP，统一鉴权与代理 |
| MCP 解耦 | 独立进程 + 独立 venv | 与 Flask 隔离，token→RBAC 复用平台权限 |
| 工作区 | 仓库外 + 每会话隔离 | 防 Agent 内置工具越权读其它会话/仓库 |
| 定时流水线 | 复用批任务引擎 | 一套并发/认领/超时机制，扫描=batch、记录=子会话 |
| Agent/模型 | 可配置（下拉/字段） | 不锁定单一模型，部署可换 provider |
