# 系统设计文档

## 1. 系统概述

巡检用例管理系统是一个基于动态配置的数据管理平台。核心设计理念是**配置驱动**：管理员通过菜单管理和页面配置定义业务数据结构，系统自动渲染对应的表单和表格，无需编写前端代码即可扩展新的业务页面。

### 技术架构

```
┌──────────────────────────────────────────┐
│              浏览器 (Vue 3)               │
│  Element Plus + Pinia + Vue Router       │
└──────────────┬───────────────────────────┘
               │ HTTP (JWT Bearer Token)
               │ Vite Proxy /api → :3001
┌──────────────▼───────────────────────────┐
│           Flask 后端 (:3001)              │
│  Blueprint 路由 + 认证装饰器              │
└──────────────┬───────────────────────────┘
               │ psycopg2 连接池
┌──────────────▼───────────────────────────┐
│         PostgreSQL 数据库                 │
│  8 张表 + JSONB 灵活存储                  │
└──────────────────────────────────────────┘
```

## 2. 数据库设计

### 2.1 表结构总览

系统共 8 张表，分为三类：

| 分类 | 表名 | 用途 |
|------|------|------|
| **业务核心** | menus | 菜单层级结构 |
| | page_configs | 页面字段配置（数据 Schema） |
| | dynamic_data | 所有业务数据的统一存储 |
| | data_relations | 多对多关联关系 |
| **系统管理** | users | 用户账号和角色 |
| | operation_logs | 操作审计日志 |
| **备份** | backups | 备份元数据 |
| | backup_settings | 定时备份配置（单行表） |

### 2.2 各表详细结构

#### menus — 菜单配置

```sql
id          VARCHAR(100) PRIMARY KEY    -- 如 menu-1, menu-2-3-1
name        VARCHAR(200) NOT NULL       -- 显示名称
icon        VARCHAR(100)                -- Element Plus 图标名
page_id     VARCHAR(100)                -- 关联的页面配置 ID（叶子菜单）
parent_id   VARCHAR(100)                -- 父菜单 ID（顶级为 NULL）
"order"     INTEGER NOT NULL DEFAULT 0  -- 排序序号
path        VARCHAR(500)                -- 路由路径
roles       JSONB DEFAULT '["admin","developer","guest"]'  -- 可见角色列表
```

#### page_configs — 页面配置

```sql
id              VARCHAR(100) PRIMARY KEY    -- 如 page-inspection-case
name            VARCHAR(200) NOT NULL       -- 页面名称
description     TEXT                        -- 页面描述
api_endpoint    VARCHAR(500)                -- API 端点路径
fields          JSONB DEFAULT '[]'          -- 字段配置数组（核心）
created_at      TIMESTAMPTZ DEFAULT NOW()
updated_at      TIMESTAMPTZ DEFAULT NOW()
```

`fields` JSONB 数组中每个元素的结构：

```json
{
  "id": "field-1",
  "label": "用例名称",
  "fieldName": "caseName",
  "controlType": "text",
  "required": true,
  "order": 1,
  "placeholder": "请输入用例名称",
  "isPrimaryKey": false,
  "options": [],
  "optionsSource": { "type": "static" },
  "relationConfig": null,
  "referenceConfig": null
}
```

#### dynamic_data — 动态业务数据

```sql
id          VARCHAR(100) PRIMARY KEY    -- 记录 ID
collection  VARCHAR(200) NOT NULL       -- 集合名（对应 page_config 去掉 page- 前缀）
data        JSONB DEFAULT '{}'          -- 业务数据（字段由 page_config 定义）
created_at  TIMESTAMPTZ DEFAULT NOW()
```

索引：`idx_dynamic_data_collection ON (collection)`

#### data_relations — 关联关系

```sql
collection          VARCHAR(200) NOT NULL   -- 源集合
record_id           VARCHAR(100) NOT NULL   -- 源记录 ID
field_name          VARCHAR(200) NOT NULL   -- 源字段名
related_collection  VARCHAR(200) NOT NULL   -- 目标集合
related_id          VARCHAR(100) NOT NULL   -- 目标记录 ID
PRIMARY KEY (collection, record_id, field_name, related_id)
```

索引：`idx_data_relations_reverse ON (related_collection, related_id)`

#### users — 用户

```sql
id              VARCHAR(100) PRIMARY KEY
username        VARCHAR(100) UNIQUE NOT NULL
password_hash   VARCHAR(256) NOT NULL       -- Werkzeug bcrypt 哈希
display_name    VARCHAR(200) NOT NULL
role            VARCHAR(50) CHECK (role IN ('admin', 'developer', 'guest'))
created_at      TIMESTAMPTZ DEFAULT NOW()
```

#### operation_logs — 操作日志

```sql
id              VARCHAR(100) PRIMARY KEY
action          VARCHAR(50) NOT NULL        -- create / update / delete
target_type     VARCHAR(100) NOT NULL       -- menu / page_config / dynamic_data / user / relation
target_id       VARCHAR(100)
target_name     VARCHAR(500)
description     VARCHAR(1000) NOT NULL      -- 中文描述
operator_id     VARCHAR(100) NOT NULL
operator_name   VARCHAR(200) NOT NULL
operator_role   VARCHAR(50) NOT NULL
created_at      TIMESTAMPTZ DEFAULT NOW()
```

索引：`created_at DESC`、`action`、`target_type`

#### backups — 备份记录

```sql
id              VARCHAR(100) PRIMARY KEY
name            VARCHAR(500) NOT NULL       -- 如 "手动备份 2026-02-14 01:00:00"
type            VARCHAR(50) DEFAULT 'manual'  -- manual / scheduled
status          VARCHAR(50) DEFAULT 'completed'
file_path       VARCHAR(1000)               -- ZIP 文件绝对路径
file_size       BIGINT DEFAULT 0
tables_count    INTEGER DEFAULT 0
records_count   INTEGER DEFAULT 0
created_by      VARCHAR(200)
created_at      TIMESTAMPTZ DEFAULT NOW()
note            TEXT
```

#### backup_settings — 定时备份配置

```sql
id              INTEGER PRIMARY KEY CHECK (id = 1)  -- 单行约束
enabled         BOOLEAN DEFAULT FALSE
interval        VARCHAR(50) DEFAULT 'daily'         -- daily / weekly / monthly
retention_count INTEGER DEFAULT 10
last_backup_at  TIMESTAMPTZ
updated_at      TIMESTAMPTZ DEFAULT NOW()
```

## 3. 数据依赖关系

### 3.1 实体关系总图

```
                    ┌──────────┐
                    │  users   │
                    │ (用户表)  │
                    └────┬─────┘
                         │ operator_id / operator_name
                         ▼
                  ┌──────────────┐
                  │operation_logs│
                  │  (操作日志)   │
                  └──────────────┘

  ┌──────────┐  page_id    ┌──────────────┐  定义 Schema   ┌──────────────┐
  │  menus   │────────────▶│ page_configs  │──────────────▶│ dynamic_data │
  │ (菜单表)  │             │  (页面配置)   │               │  (业务数据)   │
  └──────────┘             └──────────────┘               └──────┬───────┘
       │                                                         │
       │ parent_id                                    ┌──────────┴──────────┐
       │ (自引用)                                     │                     │
       ▼                                       relation 字段          reference 字段
  ┌──────────┐                                 (多对多)               (一对多依赖)
  │  menus   │                                      │                     │
  │ (父菜单)  │                                      ▼                     ▼
  └──────────┘                              ┌───────────────┐    ┌──────────────┐
                                            │data_relations │    │ dynamic_data │
                                            │ (关联关系表)   │    │  (父记录)     │
                                            └───────────────┘    └──────────────┘
```

### 3.2 菜单 → 页面配置 → 数据集合（配置继承链）

这是系统最核心的数据流转关系：

```
menus.page_id = "page-inspection-case"
       │
       ▼
page_configs.id = "page-inspection-case"
  └─ fields: [{fieldName: "caseName", controlType: "text"}, ...]
       │
       │  collection = page_id 去掉 "page-" 前缀
       ▼
dynamic_data WHERE collection = "inspection-case"
  └─ data: {"caseName": "服务器日常巡检", "caseType": "hardware", ...}
```

**依赖规则：**
- 菜单通过 `page_id` 软引用页面配置（应用层约束，非外键）
- 页面配置的 `fields` 定义了数据集合的 Schema
- 动态数据的 `collection` 与页面配置 ID 通过命名约定关联
- 删除页面配置时，已有动态数据不会被级联删除（需手动清理）

### 3.3 菜单自引用（树形层级）

```
menu-2 (巡检管理, parent_id=NULL)
  ├── menu-2-1 (巡检用例, parent_id="menu-2")
  ├── menu-2-2 (巡检计划, parent_id="menu-2")
  └── menu-2-3 (巡检记录, parent_id="menu-2")
        ├── menu-2-3-1 (日常巡检, parent_id="menu-2-3")
        └── menu-2-3-2 (专项巡检, parent_id="menu-2-3")
```

- 最多支持 3 级嵌套
- 顶级菜单 `parent_id = NULL`
- 只有叶子菜单设置 `page_id`（关联具体数据页）
- 删除父菜单时需先删除所有子菜单

### 3.4 数据关联（Relation — 多对多）

relation 字段用于建立两个数据集合之间的多对多关系，通过 `data_relations` 表存储。

```
示例：巡检计划 ←→ 巡检用例（多对多）

page_configs["page-inspection-plan"].fields 中包含:
  {
    controlType: "relation",
    fieldName: "cases",
    relationConfig: {
      targetCollection: "inspection-case",
      displayField: "caseName",
      targetField: "plans"        ← 反向字段名
    }
  }
```

**存储结构（data_relations 表）：**

```
正向: (inspection-plan, plan-1, cases, inspection-case, case-1)
正向: (inspection-plan, plan-1, cases, inspection-case, case-2)
反向: (inspection-case, case-1, plans, inspection-plan, plan-1)
反向: (inspection-case, case-2, plans, inspection-plan, plan-1)
```

**双向同步算法：**

当更新 plan-1 的 cases 关联为 [case-1, case-3] 时：
1. 查询旧关联 → [case-1, case-2]
2. 删除正向记录
3. 插入新正向记录 (plan-1 → case-1, plan-1 → case-3)
4. 计算差集：移除 = {case-2}，新增 = {case-3}
5. 删除 case-2 的反向记录
6. 插入 case-3 的反向记录

**级联删除：** 删除某条动态数据记录时，同时删除 `data_relations` 中该记录的所有正向和反向关联。

### 3.5 数据引用（Reference — 一对多依赖）

reference 字段用于建立父子依赖关系，值直接存储在 `dynamic_data.data` JSONB 中。

```
示例：日常巡检记录 → 巡检用例（多条记录引用同一父记录）

page_configs["page-daily-record"].fields 中包含:
  {
    controlType: "reference",
    fieldName: "caseRef",
    referenceConfig: {
      targetCollection: "inspection-case",
      displayField: "caseName",
      inheritFields: ["caseType", "priority"]  ← 表格中继承显示的字段
    }
  }
```

**存储方式：**

```json
// dynamic_data 中 daily-record 集合的一条记录
{
  "collection": "daily-record",
  "data": {
    "inspectionDate": "2026-01-20",
    "caseRef": "case-1",           ← 直接存储父记录 ID
    "result": "normal"
  }
}
```

**删除保护（核心约束）：**

当尝试删除 `case-1` 时，系统会检查所有页面配置中的 reference 字段：

```python
# server/routes/dynamic.py - check_reference_dependencies()
遍历所有 page_configs:
  对每个 controlType="reference" 且 targetCollection="inspection-case" 的字段:
    查询: SELECT count(*) FROM dynamic_data
           WHERE collection='daily-record' AND data->>'caseRef' = 'case-1'
    如果 count > 0:
      返回 409 错误: "无法删除：被「日常巡检」的 N 条记录引用"
```

**Relation 与 Reference 的区别：**

| 特性 | Relation（关联） | Reference（引用） |
|------|-----------------|-----------------|
| 关系类型 | 多对多 | 一对多（多个子记录引用同一父记录） |
| 存储位置 | data_relations 表 | dynamic_data.data JSONB 字段 |
| 双向同步 | 自动维护正向+反向 | 单向（子→父） |
| 删除保护 | 无（级联清理关联记录） | 有（父记录被引用时禁止删除） |
| 继承显示 | 无 | 支持在表格中显示父记录的字段 |
| 值类型 | 多个 ID 数组 | 单个 ID |

### 3.6 菜单角色可见性

```
menus.roles = ["admin", "developer"]
                    ↓ 与用户角色取交集
users.role = "developer"
                    ↓
交集非空 → 菜单可见
```

前端侧边栏根据当前用户角色过滤菜单树；后端路由守卫二次校验路径权限。

### 3.7 主键唯一性约束

页面配置中可将字段标记为 `isPrimaryKey: true`，系统在新增/修改数据时会检查主键组合的唯一性：

```python
# server/routes/dynamic.py - check_primary_key_unique()
查询: SELECT id FROM dynamic_data
      WHERE collection = %s AND data->>pk_field1 = %s AND data->>pk_field2 = %s
如果存在 → 返回 409 错误: "主键重复：field1=value1, field2=value2"
```

支持复合主键（多个字段同时标记为主键）。

## 4. 认证与权限模型

### 4.1 JWT 认证流程

```
登录请求 (username + password)
       │
       ▼
验证密码 (werkzeug.check_password_hash)
       │
       ▼
生成 JWT Token (payload: userId, username, role, exp)
       │
       ▼
前端存储 Token (localStorage: check-manage:token)
       │
       ▼
后续请求携带 Authorization: Bearer {token}
       │
       ▼
后端装饰器解码验证 → g.current_user = {userId, username, role}
```

### 4.2 角色权限矩阵

| 功能 | admin | developer | guest |
|------|:-----:|:---------:|:-----:|
| 查看数据页 | ✓ | ✓ | ✓ |
| 新增/编辑/删除数据 | ✓ | ✓ | ✓ |
| 导入/导出 Excel | ✓ | ✓ | ✓ |
| 菜单管理 | ✓ | | |
| 页面配置 | ✓ | | |
| 用户管理 | ✓ | | |
| 操作日志 | ✓ | | |
| 系统备份 | ✓ | | |

菜单可见性额外受 `menus.roles` 字段控制，管理员可在菜单管理中为每个菜单单独设置可见角色。

### 4.3 后端权限装饰器

- `@login_required` — 所有已认证用户可访问
- `@admin_required` — 仅 admin 角色可访问

## 6. 备份与还原

### 6.1 备份内容

备份覆盖 6 张业务表（不含 backups 和 backup_settings 自身）：

```
menus, page_configs, dynamic_data, data_relations, users, operation_logs
```

### 6.2 备份格式

ZIP 文件内部结构：

```
manifest.json           ← 版本号、时间戳、各表记录数
menus.json
page_configs.json
dynamic_data.json
data_relations.json
users.json
operation_logs.json
```

`manifest.json` 包含 `version` 字段用于未来数据迁移兼容。

### 6.3 还原机制

在单个数据库事务中执行：
1. TRUNCATE 全部 6 张业务表（CASCADE）
2. 从 JSON 文件逐条 INSERT（JSONB 列自动转换）
3. 成功则 COMMIT，任何异常则 ROLLBACK

还原不影响 backups 和 backup_settings 表。

### 6.4 定时备份

后台守护线程每 60 秒轮询 `backup_settings` 表：
- 若 `enabled = true` 且距 `last_backup_at` 超过设定间隔 → 触发备份
- 备份完成后自动清理超出 `retention_count` 的旧定时备份

Flask debug 模式下仅在 reloader 子进程中启动调度器，避免重复执行。

## 7. 前端架构

### 7.1 路由结构

```
/login                    公开页面
/                         主布局（需认证）
├── /home                 首页（统计卡片 + 快捷入口）
├── /page/:pageId         动态数据页（根据配置自动渲染）
├── /admin/menu           菜单管理
├── /admin/page-config    页面配置管理
├── /admin/users          用户管理
├── /admin/operation-log  操作日志
└── /admin/backup         系统备份
```

动态数据页路由在应用初始化时根据菜单配置自动注册。

### 7.2 状态管理（Pinia Stores）

| Store | 职责 |
|-------|------|
| authStore | JWT Token、用户信息、路由权限判断 |
| appStore | 应用初始化状态 |
| menuStore | 菜单树构建、菜单 CRUD |
| pageConfigStore | 页面配置 CRUD、字段管理 |

### 7.3 支持的字段控件类型

| 控件类型 | 说明 | 存储类型 |
|---------|------|---------|
| text | 单行文本 | string |
| textarea | 多行文本 | string |
| number | 数值输入 | number |
| select | 单选下拉 | string |
| multiSelect | 多选下拉 | string[] |
| radio | 单选按钮组 | string |
| checkbox | 复选框组 | string[] |
| date | 日期选择 | string (YYYY-MM-DD) |
| datetime | 日期时间选择 | string (YYYY-MM-DD HH:mm:ss) |
| file | 文件上传 | object[] |
| image | 图片上传 | object[] |
| relation | 多对多关联 | 通过 data_relations 表 |
| reference | 一对多引用 | string (父记录 ID) |

## 9. 扩展性设计

### 新增业务数据页

无需修改代码，仅需配置：
1. 在页面配置中创建新页面，定义字段
2. 在菜单管理中创建菜单项，关联页面配置
3. 系统自动渲染对应的表单和表格

### 新增数据库表

若未来需要新增数据库表：
1. 在 `init_db.py` DDL 中追加建表语句
2. 在 `backup.py` 的 `BACKUP_TABLES` 列表中追加表定义
3. 备份/还原自动覆盖新表

### 动态路由保护

`server/routes/dynamic.py` 中的 `RESERVED` 集合维护了系统保留路径名，新增 API 蓝图时需在此集合中注册，避免被动态路由拦截：

```python
RESERVED = {'menus', 'pageConfigs', 'favicon.ico', 'relations',
            'auth', 'users', 'operationLogs', 'backups'}
```
