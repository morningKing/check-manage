# 系统设计文档

## 1. 系统概述

巡检用例管理系统是一个基于动态配置的数据管理平台。核心设计理念是**配置驱动**：管理员通过菜单管理和页面配置定义业务数据结构，系统自动渲染对应的表单和表格，无需编写前端代码即可扩展新的业务页面。

### 技术架构

```
┌──────────────────────────────────────────┐
│              浏览器 (Vue 3)               │
│  Element Plus + Pinia + Vue Router       │
└──────────────┬───────────────────────────┘
               │ HTTP (JWT Bearer Token)
               │ Vite Proxy /api → :3001
┌──────────────▼───────────────────────────┐
│           Flask 后端 (:3001)              │
│  Blueprint 路由 + 认证装饰器              │
└──────────────┬───────────────────────────┘
               │ psycopg2 连接池
┌──────────────▼───────────────────────────┐
│         PostgreSQL 数据库                 │
│  13 张表 + JSONB 灵活存储                 │
└──────────────────────────────────────────┘
```

## 2. 数据库设计

### 2.1 表结构总览

系统共 13 张业务表（另有 2 张备份管理表），分为六类：

| 分类 | 表名 | 用途 |
|------|------|------|
| **业务核心** | menus | 菜单层级结构 |
| | page_configs | 页面字段配置（数据 Schema） |
| | dynamic_data | 所有业务数据的统一存储 |
| | data_relations | 多对多关联关系 |
| **脚本引擎** | export_scripts | 自定义导出脚本 |
| | validation_scripts | 数据校验脚本 |
| **ETL 数据管道** | etl_tasks | ETL 任务定义（含管道步骤） |
| | etl_logs | ETL 执行日志 |
| **对外接口** | api_keys | Open API 密钥管理 |
| **系统管理** | users | 用户账号和角色 |
| | operation_logs | 操作审计日志（支持批次分组） |
| **备份** | backups | 备份元数据 |
| | backup_settings | 定时备份配置（单行表） |

### 2.2 各表详细结构

#### menus — 菜单配置

```sql
id          VARCHAR(100) PRIMARY KEY    -- 如 menu-1, menu-2-3-1
name        VARCHAR(200) NOT NULL       -- 显示名称
icon        VARCHAR(100)                -- Element Plus 图标名
page_id     VARCHAR(100)                -- 关联的页面配置 ID（叶子菜单）
parent_id   VARCHAR(100)                -- 父菜单 ID（顶级为 NULL）
"order"     INTEGER NOT NULL DEFAULT 0  -- 排序序号
path        VARCHAR(500)                -- 路由路径
roles       JSONB DEFAULT '["admin","developer","guest"]'  -- 可见角色列表
```

#### page_configs — 页面配置

```sql
id                  VARCHAR(100) PRIMARY KEY    -- 如 page-inspection-case
name                VARCHAR(200) NOT NULL       -- 页面名称
description         TEXT                        -- 页面描述
api_endpoint        VARCHAR(500)                -- API 端点路径
fields              JSONB DEFAULT '[]'          -- 字段配置数组（核心）
export_scripts      JSONB DEFAULT '[]'          -- 绑定的页面级导出脚本 ID 列表
row_export_scripts  JSONB DEFAULT '[]'          -- 绑定的行级导出脚本 ID 列表
api_public          BOOLEAN DEFAULT FALSE       -- 是否开放 Open API 访问
validation_script   TEXT                        -- 绑定的校验脚本 ID
created_at          TIMESTAMPTZ DEFAULT NOW()
updated_at          TIMESTAMPTZ DEFAULT NOW()
```

`fields` JSONB 数组中每个元素的结构：

```json
{
  "id": "field-1",
  "label": "用例名称",
  "fieldName": "caseName",
  "controlType": "text",
  "required": true,
  "order": 1,
  "placeholder": "请输入用例名称",
  "isPrimaryKey": false,
  "options": [],
  "optionsSource": { "type": "static" },
  "relationConfig": null,
  "referenceConfig": null
}
```

#### dynamic_data — 动态业务数据

```sql
id          VARCHAR(100) PRIMARY KEY    -- 记录 ID
collection  VARCHAR(200) NOT NULL       -- 集合名（对应 page_config 去掉 page- 前缀）
data        JSONB DEFAULT '{}'          -- 业务数据（字段由 page_config 定义）
created_at  TIMESTAMPTZ DEFAULT NOW()
```

索引：`idx_dynamic_data_collection ON (collection)`

#### data_relations — 关联关系

```sql
collection          VARCHAR(200) NOT NULL   -- 源集合
record_id           VARCHAR(100) NOT NULL   -- 源记录 ID
field_name          VARCHAR(200) NOT NULL   -- 源字段名
related_collection  VARCHAR(200) NOT NULL   -- 目标集合
related_id          VARCHAR(100) NOT NULL   -- 目标记录 ID
PRIMARY KEY (collection, record_id, field_name, related_id)
```

索引：`idx_data_relations_reverse ON (related_collection, related_id)`

#### export_scripts — 导出脚本

```sql
id              VARCHAR(100) PRIMARY KEY    -- 如 script-abc123
name            VARCHAR(200) NOT NULL       -- 脚本名称
description     TEXT                        -- 描述说明
language        VARCHAR(50) DEFAULT 'python'-- 脚本语言
script          TEXT NOT NULL               -- 脚本代码
output_format   VARCHAR(50) DEFAULT 'json'  -- 输出格式 (json/csv/xml/txt/html)
scope           VARCHAR(50) DEFAULT 'page'  -- 导出维度: page=整页 / row=单行
created_at      TIMESTAMPTZ DEFAULT NOW()
updated_at      TIMESTAMPTZ DEFAULT NOW()
```

#### validation_scripts — 校验脚本

```sql
id              VARCHAR(100) PRIMARY KEY    -- 如 vs-abc12345
name            VARCHAR(200) NOT NULL       -- 脚本名称
description     TEXT                        -- 描述说明
script          TEXT NOT NULL DEFAULT ''    -- Python 脚本代码
created_at      TIMESTAMPTZ DEFAULT NOW()
updated_at      TIMESTAMPTZ DEFAULT NOW()
```

#### etl_tasks — ETL 任务

```sql
id              VARCHAR(100) PRIMARY KEY    -- 如 etl-abc123def456
name            VARCHAR(200) NOT NULL       -- 任务名称
description     TEXT                        -- 任务描述
steps           JSONB DEFAULT '[]'          -- 管道步骤配置（核心）
enabled         BOOLEAN DEFAULT TRUE        -- 是否启用
last_run_at     TIMESTAMPTZ                 -- 最近执行时间
last_run_status VARCHAR(50)                 -- 最近执行状态 (success/partial/error)
created_at      TIMESTAMPTZ DEFAULT NOW()
updated_at      TIMESTAMPTZ DEFAULT NOW()
```

`steps` JSONB 数组中每个步骤的结构：

```json
{
  "id": "step-xxx",
  "type": "http_request",
  "name": "获取外部数据",
  "config": { ... },
  "onError": "stop"
}
```

#### etl_logs — ETL 执行日志

```sql
id              VARCHAR(100) PRIMARY KEY
task_id         VARCHAR(100) NOT NULL       -- 关联的任务 ID
task_name       VARCHAR(200)                -- 任务名称快照
status          VARCHAR(50) NOT NULL        -- success / partial / error
started_at      TIMESTAMPTZ NOT NULL
finished_at     TIMESTAMPTZ
total_records   INTEGER DEFAULT 0
success_count   INTEGER DEFAULT 0
error_count     INTEGER DEFAULT 0
step_results    JSONB DEFAULT '[]'          -- 各步骤执行结果
error_detail    TEXT                        -- 错误详情
created_at      TIMESTAMPTZ DEFAULT NOW()
```

索引：`idx_etl_logs_task_id`、`idx_etl_logs_created_at DESC`

#### api_keys — API 密钥

```sql
id              VARCHAR(100) PRIMARY KEY    -- 如 ak-abc12345
name            VARCHAR(200) NOT NULL       -- 密钥名称
key_hash        VARCHAR(256) NOT NULL       -- 密钥哈希（Werkzeug SHA256）
created_at      TIMESTAMPTZ DEFAULT NOW()
last_used_at    TIMESTAMPTZ                 -- 最近使用时间
is_active       BOOLEAN DEFAULT TRUE        -- 是否启用
```

#### users — 用户

```sql
id              VARCHAR(100) PRIMARY KEY
username        VARCHAR(100) UNIQUE NOT NULL
password_hash   VARCHAR(256) NOT NULL       -- Werkzeug bcrypt 哈希
display_name    VARCHAR(200) NOT NULL
role            VARCHAR(50) CHECK (role IN ('admin', 'developer', 'guest'))
created_at      TIMESTAMPTZ DEFAULT NOW()
```

#### operation_logs — 操作日志

```sql
id              VARCHAR(100) PRIMARY KEY
action          VARCHAR(50) NOT NULL        -- create / update / delete
target_type     VARCHAR(100) NOT NULL       -- menu / page_config / dynamic_data / ...
target_id       VARCHAR(100)
target_name     VARCHAR(500)
description     VARCHAR(1000) NOT NULL      -- 中文描述
operator_id     VARCHAR(100) NOT NULL
operator_name   VARCHAR(200) NOT NULL
operator_role   VARCHAR(50) NOT NULL
batch_id        VARCHAR(100)                -- 批次 ID
batch_desc      VARCHAR(500)                -- 批次描述
created_at      TIMESTAMPTZ DEFAULT NOW()
```

索引：`created_at DESC`、`action`、`target_type`、`batch_id`

#### backups / backup_settings

```sql
-- backups
id VARCHAR(100) PRIMARY KEY, name, type, status, file_path, file_size,
tables_count, records_count, created_by, created_at, note

-- backup_settings（单行表，id 固定为 1）
id INTEGER PRIMARY KEY CHECK (id = 1), enabled BOOLEAN, interval VARCHAR(50),
retention_count INTEGER, last_backup_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
```

### 3.2 菜单 → 页面配置 → 数据集合（配置继承链）

这是系统最核心的数据流转关系：

```
menus.page_id = "page-inspection-case"
       │
       ▼
page_configs.id = "page-inspection-case"
  ├─ fields: [{fieldName: "caseName", controlType: "text"}, ...]
  ├─ validation_script: "vs-abc12345"  →  validation_scripts 表
  ├─ export_scripts: ["script-xyz"]    →  export_scripts 表
  └─ api_public: true                  →  Open API 可访问
       │
       │  collection = page_id 去掉 "page-" 前缀
       ▼
dynamic_data WHERE collection = "inspection-case"
  └─ data: {"caseName": "服务器日常巡检", "caseType": "hardware", ...}
```

### 3.3 菜单层级结构

系统菜单分为三级，其中系统配置菜单进一步按功能分为三个子组：

```
menu-1 首页                              (admin, developer, guest)
menu-2 巡检管理                          (admin, developer)
  ├── menu-2-1 巡检用例                  → page-inspection-case
  ├── menu-2-2 巡检计划                  → page-inspection-plan
  └── menu-2-3 巡检记录
        ├── menu-2-3-1 日常巡检          → page-daily-record
        └── menu-2-3-2 专项巡检          → page-special-record
menu-3 系统配置                          (admin)
  ├── menu-3-a 平台管理
  │     ├── menu-3-1 菜单管理            /admin/menu
  │     ├── menu-3-2 页面配置            /admin/page-config
  │     └── menu-3-3 用户管理            /admin/users
  ├── menu-3-b 数据工具
  │     ├── menu-3-6 导出脚本            /admin/export-scripts
  │     ├── menu-3-8 校验脚本            /admin/validation-scripts
  │     ├── menu-3-7 Open API            /admin/api-keys
  │     └── menu-3-9 ETL 管理            /admin/etl-tasks
  └── menu-3-c 系统运维
        ├── menu-3-4 操作日志            /admin/operation-log
        └── menu-3-5 系统备份            /admin/backup
```

**内置菜单保护：**
- 首页（menu-1）和系统配置整棵树（menu-3 及其所有后代）均为内置菜单
- 后端强制校验：内置菜单不允许编辑和删除，系统配置树下不允许添加子菜单
- 前端展示锁图标，编辑表单自动置为只读

### 3.4 数据关联（Relation — 多对多）

relation 字段用于建立两个数据集合之间的多对多关系，通过 `data_relations` 表存储。

**双向同步算法：** 当更新 plan-1 的 cases 关联为 [case-1, case-3] 时：
1. 查询旧关联 → [case-1, case-2]
2. 删除正向记录
3. 插入新正向记录 (plan-1 → case-1, plan-1 → case-3)
4. 计算差集：移除 = {case-2}，新增 = {case-3}
5. 删除 case-2 的反向记录
6. 插入 case-3 的反向记录

**级联删除：** 删除某条动态数据记录时，同时清理 `data_relations` 中该记录的所有正向和反向关联。

### 3.5 数据引用（Reference — 一对多依赖）

reference 字段值直接存储在 `dynamic_data.data` JSONB 中，父记录 ID 作为字段值。

**删除保护：** 尝试删除被引用的父记录时，系统遍历所有 page_configs 的 reference 字段进行反向查询，存在引用则返回 409 错误。

| 特性 | Relation（关联） | Reference（引用） |
|------|-----------------|-----------------|
| 关系类型 | 多对多 | 一对多 |
| 存储位置 | data_relations 表 | dynamic_data.data JSONB |
| 双向同步 | 自动维护正向+反向 | 单向（子→父） |
| 删除保护 | 无（级联清理） | 有（禁止删除被引用的父记录） |

### 3.6 校验脚本绑定

```
page_configs.validation_script = "vs-abc12345"  (校验脚本 ID)
       │
       ▼
validation_scripts.id = "vs-abc12345"
  └─ script: "if not record.get('name'): add_error('名称不能为空')"
       │
       │  在 dynamic_data CREATE / UPDATE 时自动执行
       ▼
run_validation_script(script, record, action, old_data, fields, collection, conn)
  └─ 返回 (errors, warnings, pending_relations)
```

脚本可通过 `add_error()` 阻止保存，通过 `set_relations()` 排队关联操作。

### 3.7 导出脚本绑定

```
page_configs.export_scripts = ["script-abc"]       ← 页面级导出脚本
page_configs.row_export_scripts = ["script-def"]   ← 行级导出脚本
```

删除导出脚本时自动清理所有页面配置中对该脚本的引用。

## 4. 认证与权限模型

### 4.1 JWT 认证流程

```
登录请求 (username + password)
  → 验证密码 (werkzeug.check_password_hash)
  → 生成 JWT Token (payload: userId, username, role, exp)
  → 前端存储 Token (localStorage)
  → 后续请求携带 Authorization: Bearer {token}
  → 后端装饰器解码验证 → g.current_user
```

### 4.2 角色权限矩阵

| 功能 | admin | developer | guest |
|------|:-----:|:---------:|:-----:|
| 查看数据页 | Y | Y | Y |
| 新增/编辑/删除数据 | Y | Y | Y |
| 导入/导出 | Y | Y | Y |
| 菜单管理 | Y | | |
| 页面配置 | Y | | |
| 用户管理 | Y | | |
| 操作日志 | Y | | |
| 系统备份 | Y | | |
| 导出脚本管理 | Y | | |
| 校验脚本管理 | Y | | |
| Open API 管理 | Y | | |
| ETL 管理 | Y | | |

### 4.3 Open API 认证

外部系统通过 API Key 访问公开数据集合：

```
请求携带 X-API-Key: cm_xxxxxxxxxx
  → 查询 api_keys 表，匹配 key_hash
  → 校验 is_active = true
  → 更新 last_used_at
  → 仅允许访问 api_public = true 的数据集合
```

API Key 格式：`cm_<32位URL安全随机字符>`，创建时明文显示一次，后续仅存储哈希值。

### 4.4 后端权限装饰器

- `@login_required` — 所有已认证用户可访问
- `@admin_required` — 仅 admin 角色可访问
- `@api_key_required` — Open API 密钥认证

## 5. 脚本执行沙箱

导出脚本、校验脚本和 ETL 脚本共享同一个沙箱执行引擎 (`server/utils/script_runner.py`)。

### 5.1 安全模型

```
1. 语法检查（正则）
   - 禁止 import / from...import 语句
   - 禁止 open(), exec(), eval(), compile() 等危险函数
   - 禁止 __dunder__ 属性访问

2. 受限全局
   - 仅注入白名单模块：json, csv, io, re, math, collections, xml.etree, datetime
   - 仅注入安全内置函数：len, str, int, float, list, dict, sorted, range 等 20+

3. 超时保护
   - 独立线程执行，10 秒超时自动终止
```

### 5.2 三种脚本类型

| 类型 | 函数 | 注入变量 | 输出要求 |
|------|------|---------|---------|
| 导出脚本 | `run_export_script()` | data, fields, page_name | result (str/bytes) |
| 校验脚本 | `run_validation_script()` | record, action, old_data, fields, collection + 辅助函数 | add_error()/add_warning() |
| ETL 脚本 | `run_etl_script()` | records | result (list[dict]) |

校验脚本额外提供数据库查询辅助函数：`query()`, `query_one()`, `find_by()`, `get_relations()`, `set_relations()`。

## 6. ETL 数据管道

### 6.1 架构概述

ETL 执行引擎 (`server/utils/etl_engine.py`) 按顺序执行管道步骤，数据在步骤间通过 `context['records']` 流转。

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ HTTP 请求 │───▶│ 脚本转换 │───▶│ 字段映射 │───▶│ 写入集合 │
│ (数据源)  │    │ (清洗)   │    │ (重命名) │    │ (目标)   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     records ────▶ records ────▶ records ────▶ records
```

### 6.2 步骤类型

| 类型 | 标识 | 说明 |
|------|------|------|
| HTTP 请求 | `http_request` | 调用外部 API 获取 JSON 数据，支持 GET/POST、自定义 Headers、响应路径提取 |
| JSON 输入 | `json_input` | 手动输入 JSON 数据，用于测试或少量数据导入 |
| Python 脚本 | `script` | 自定义转换逻辑，注入 `records`，须设置 `result` |
| 字段映射 | `field_mapping` | 源字段→目标字段重命名，可选保留未映射字段 |
| 条件过滤 | `filter` | Python 表达式过滤记录，`record` 变量指向当前记录 |
| 写入集合 | `save_to_collection` | 将数据写入 dynamic_data 表，支持 insert/upsert/update 三种模式 |

### 6.3 写入模式

| 模式 | 说明 |
|------|------|
| insert | 全部新增，自动生成 ID |
| upsert | 按匹配字段查找，有则更新无则插入 |
| update | 按匹配字段查找，仅更新已有记录 |

### 6.4 错误处理

每个步骤可配置 `onError` 策略：
- `stop` — 出错时停止整个管道
- `skip` — 出错时跳过该步骤，继续执行后续步骤

### 6.5 Dry Run

测试运行模式 (`dryRun: true`) 在独立事务中执行全部步骤，完成后 ROLLBACK，不实际写入数据库。用于验证管道配置的正确性。

## 7. 备份与还原

### 7.1 备份内容

备份覆盖 11 张业务表：

```
menus, page_configs, dynamic_data, data_relations, users, operation_logs,
export_scripts, api_keys, validation_scripts, etl_tasks, etl_logs
```

### 7.2 备份格式

ZIP 文件内含 `manifest.json`（版本号、时间戳、各表记录数）+ 各表对应的 JSON 文件。

### 7.3 还原机制

在单个数据库事务中执行：TRUNCATE 全部业务表（CASCADE）→ 逐条 INSERT → 成功 COMMIT / 失败 ROLLBACK。

### 7.4 定时备份

后台守护线程每 60 秒轮询 `backup_settings` 表，到期自动执行备份并清理超出保留数量的旧备份。

## 10. 扩展性设计

### 新增业务数据页

无需修改代码，仅需配置：
1. 页面配置中创建新页面，定义字段
2. 菜单管理中创建菜单项，关联页面配置
3. 系统自动渲染对应的表单和表格

### 新增自定义导出

1. 导出脚本管理中创建脚本
2. 在页面配置中绑定
3. 数据页自动出现导出入口

### 新增数据校验

1. 校验脚本管理中创建脚本
2. 在页面配置中绑定
3. 新增/修改数据时自动执行

### ETL 数据导入

1. ETL 管理中创建任务，编排管道步骤
2. 测试运行验证数据流转
3. 正式执行写入目标集合

### 动态路由保护

`server/routes/dynamic.py` 中的 `RESERVED` 集合维护系统保留路径名，新增 API 蓝图时需注册：

```python
RESERVED = {'menus', 'pageConfigs', 'favicon.ico', 'relations', 'auth',
            'users', 'operationLogs', 'backups', 'exportScripts',
            'apiKeys', 'validationScripts', 'etlTasks'}
```
