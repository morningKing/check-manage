# Check-Manage 测试用例文档

## 文档概述

本文档详细描述 Check-Manage 系统的全量测试用例，重点覆盖数据版本管理、备份还原、分支管理等核心功能，确保每个功能点都有详细的测试步骤和预期结果。

---

## 1. 版本管理系统测试用例

### 1.1 版本创建测试

#### TC-VER-001: 创建快照版本 - 正常流程

**前置条件**: 
- 用户已登录且具有 `developer` 或 `admin` 角色
- 目标 Collection 存在数据
- 用户在主分支工作

**测试步骤**:
1. 打开目标 Collection 的数据页面
2. 点击"版本管理"按钮打开版本面板
3. 点击"创建版本"按钮
4. 输入版本名称：`测试快照-v1`
5. 选择版本类型：`快照`
6. 输入描述：`测试快照创建`
7. 点击"创建"按钮

**预期结果**:
- 版本创建成功，显示成功提示
- 版本列表中新增一条记录，类型显示"快照"
- 版本状态为 `active`
- `recordsCount` 显示正确的记录数
- `relationsCount` 显示正确的关联数

**验证点**:
```sql
-- 验证版本元数据
SELECT * FROM collection_versions WHERE name = '测试快照-v1';
-- 验证快照数据
SELECT COUNT(*) FROM version_snapshots WHERE version_id = '<version_id>';
-- 验证跨 Collection 追踪
SELECT * FROM version_collections WHERE version_id = '<version_id>';
```

---

#### TC-VER-002: 创建快照 - 跨 Collection 关联数据

**前置条件**:
- 目标 Collection 有 M:N 关联到其他 Collection
- 关联的 Collection 中有数据

**测试步骤**:
1. 创建版本快照
2. 检查快照是否包含所有关联 Collection 的数据

**预期结果**:
- 快照包含起始 Collection 的所有数据
- 快照包含所有通过 M:N 关联的 Collection 数据
- `version_collections` 表记录所有涉及的 Collection
- 关联数据存储在 `version_snapshots` 和 `version_relations` 表

**验证点**:
```sql
-- 检查跨 Collection 数据
SELECT collection, COUNT(*) FROM version_snapshots 
WHERE version_id = '<version_id>' GROUP BY collection;
```

---

#### TC-VER-003: 创建快照 - 超大数据量限制

**前置条件**:
- Collection 数据量超过 10000 条
- 存在大量跨 Collection 关联

**测试步骤**:
1. 尝试创建版本快照

**预期结果**:
- 返回错误：`Snapshot size exceeds limit: X records (max 10000)`
- 不创建版本记录
- 前端显示错误提示

---

#### TC-VER-004: 创建分支版本 - 正常流程

**前置条件**:
- 用户在主分支工作
- Collection 有数据

**测试步骤**:

2. 点击"创建版本"
3. 输入版本名称：`功能分支-A`
4. 选择版本类型：`分支`

**预期结果**:
- 分支创建成功
- 版本列表显示类型为"分支"
- 分支状态为 `active`
- 快照数据正常存储

#### TC-VER-005: 创建分支 - 设置父版本

- 正确显示删除记录列表

**验证点**:
- `added` 数组包含所有新增的记录 ID
- `removed` 数组包含所有删除的记录 ID
- `modified[].fields` 显示字段级 oldValue → newValue

### 1.3 版本合并测试

#### TC-VER-012: 全量合并 - theirs 策略

**前置条件**:
- 存在版本快照
- 当前数据与快照有差异

**测试步骤**:
1. 选择一个版本
2. 点击"合并到当前"
3. 选择策略：`theirs`（使用源版本数据）
4. 确认合并

**预期结果**:
- 当前数据完全被源版本数据覆盖
- 新增的记录被插入到当前分支
- 删除的记录从当前分支移除
- 修改的记录被更新
- 关联关系被重建
- 版本状态变为 `merged`
- `merged_by` 和 `merged_at` 字段更新

**验证点**:
```sql
-- 检查版本状态
SELECT status, merged_by, merged_at FROM collection_versions WHERE id = '<version_id>';
-- 检查数据一致性
SELECT data_hash FROM collection_versions WHERE id = '<version_id>';
```

- 合并记录数为 0

- 版本无子版本

**前置条件**:
- 存在分支类型版本
- 分支有数据

1. 获取影响报告
2. 确认删除

**预期结果**:
- 清理 `dynamic_data` 中 `branch_id = <version_id>` 的数据
- 清理 `data_relations` 中对应分支的关联

**验证点**:
```sql
SELECT COUNT(*) FROM dynamic_data WHERE branch_id = '<version_id>'; -- 应为 0
SELECT COUNT(*) FROM data_relations WHERE branch_id = '<version_id>'; -- 应为 0
```

## 2. 分支管理系统测试用例

### 2.1 分支切换测试

#### TC-BR-001: 切换到分支 - 正常流程

**前置条件**:
- 存在分支类型版本
- 用户在主分支工作

**测试步骤**:
1. 选择一个分支
2. 点击"切换"按钮
3. 确认切换
4. 刷新数据页面

**预期结果**:
- 切换成功提示：`已切换到分支「xxx」，加载 X 条记录`
- 用户看到该分支的数据（而非主分支数据）
- 版本列表显示"当前"标签
- 分支切换按钮禁用（当前已在该分支）
- `user_current_branch` 表更新

**验证点**:
```sql
SELECT branch_id FROM user_current_branch WHERE user_id = '<uid>' AND collection = '<coll>';
-- 应返回分支 ID
```

---

#### TC-BR-002: 切换到分支 - 分支数据未初始化

**前置条件**:
- 分支刚创建，无独立数据
- 主分支有数据

**测试步骤**:
1. 切换到该分支

**预期结果**:
- 系统从主分支复制数据到该分支
- 提示：`（分支数据已从快照初始化）`
- 用户看到与主分支相同的数据
- 后续修改不影响主分支

**验证点**:
```sql
-- 分支数据与主分支相同
SELECT COUNT(*) FROM dynamic_data WHERE branch_id = '<branch_id>';
SELECT COUNT(*) FROM dynamic_data WHERE branch_id = 'main';
-- 数量应相等（初始状态）
```

---

#### TC-BR-003: 切换分支 - 跨 Collection 同步切换

**前置条件**:
- 分支涉及多个 Collection（有跨 Collection 关联）
- 用户在其他 Collection 也设置了分支

**测试步骤**:
1. 切换分支

**预期结果**:
- 同时切换所有涉及的 Collection
- 提示：`同时切换了 X 个关联集合`
- `affectedCollections` 返回所有涉及的 Collection

**验证点**:
```sql
SELECT collection, branch_id FROM user_current_branch WHERE user_id = '<uid>';
-- 所有涉及 Collection 应更新到同一分支
```

---

#### TC-BR-004: 切换分支后修改数据 - 隔离验证

**前置条件**:
- 已切换到分支

**测试步骤**:
1. 在分支上修改一条记录
2. 切换回主分支
3. 检查该记录状态

**预期结果**:
- 分支修改不影响主分支数据
- 主分支数据保持原状态
- 切换回分支后看到修改后的数据

---

#### TC-BR-005: 不同用户切换到不同分支 - 并行开发

**前置条件**:
- 存在多个分支
- 多用户登录

**测试步骤**:
1. 用户 A 切换到分支 A
2. 用户 B 切换到分支 B
3. 两个用户分别修改数据

**预期结果**:
- 两个用户的数据修改互不影响
- 每个用户只能看到自己分支的数据
- 切换分支后各自看到各自的修改
