# 数据关系类型使用说明

## 概述

系统支持三种数据关系类型，用于在不同数据集合之间建立关联关系：

| 关系类型 | 控件类型 | 关系方向 | 数量限制 | 存储位置 | 典型场景 |
|---------|---------|---------|---------|---------|---------|
| **关联关系** | relation | 双向自动同步 | 多对多 | 独立关联表 | 用例与模板互相关联 |
| **数据引用** | reference | 单向（父→子） | 一对多（单选） | 子记录 JSONB | 任务归属某个项目 |
| **引用选择** | quoteSelect | 单向（仅源端） | 一对多（多选） | 源记录 JSONB | 巡检记录引用多条用例 |

---

## 一、关联关系（relation）

### 1.1 特性说明

关联关系是一种**多对多双向关联**，核心特性：

- **双向同步**：在一侧添加或移除关联，另一侧自动同步
- **多对多**：两侧都可以选择多条记录建立关联
- **独立存储**：关联数据存储在 `data_relations` 表，与记录本身分离
- **自动清理**：删除记录时，关联方引用自动清除

### 1.2 适用场景

- 用例与模板互相关联：一个用例可属于多个模板，一个模板可包含多个用例
- 文档与标签双向关联：文档可打多个标签，标签下有多篇文档
- 人员与技能互相关联：人员具备多项技能，技能对应多位人员

### 1.3 配置步骤

以「巡检用例」与「巡检模板」互相关联为例。

**第一步：给巡检模板添加关联字段**

1. 进入 **系统配置 > 页面配置**
2. 在左侧列表选中「巡检模板」
3. 在右侧字段配置区域点击 **添加字段**
4. 填写字段信息：
   - **显示名称**：`关联用例`
   - **字段名**：`cases`（英文，用于数据存储）
   - **控件类型**：选择 `关联选择`
5. 底部出现**关联配置**区域，填写：
   - **目标集合**：下拉选择 `巡检用例`
   - **显示字段**：下拉选择 `用例名称`
   - **反向字段**：输入 `templates`（稍后在巡检用例侧创建的字段名）
6. 点击 **确定** 保存字段，再点击 **保存配置**

**第二步：给巡检用例添加关联字段**

1. 在左侧列表选中「巡检用例」
2. 点击 **添加字段**，填写：
   - **显示名称**：`所属模板`
   - **字段名**：`templates`（与上一步的反向字段名一致）
   - **控件类型**：选择 `关联选择`
3. 关联配置：
   - **目标集合**：下拉选择 `巡检模板`
   - **显示字段**：下拉选择 `模板名称`
   - **反向字段**：下拉选择 `关联用例`
4. 点击 **确定**，再点击 **保存配置**

> **关键点**：两侧的字段名和反向字段名必须互相对应。即 A 侧的字段名 = B 侧的反向字段名，反之亦然。

**配置对照表：**

| 配置项 | 巡检用例侧 | 巡检模板侧 |
|--------|-----------|-----------|
| 显示名称 | 所属模板 | 关联用例 |
| 字段名 | `templates` | `cases` |
| 目标集合 | 巡检模板 | 巡检用例 |
| 显示字段 | 模板名称 | 用例名称 |
| 反向字段 | `cases` | `templates` |

### 1.4 使用方式

**新增记录时选择关联：**

1. 进入任一数据页（如「巡检用例」），点击 **新增**
2. 填写普通字段后，在「所属模板」关联字段中勾选要关联的模板记录
3. 点击 **确定** 提交

此时：
- 巡检用例记录保存成功，其「所属模板」字段显示所选模板数量
- 切换到「巡检模板」页面，对应模板记录的「关联用例」字段**自动包含**刚创建的用例

**编辑记录时修改关联：**

1. 点击某条记录的 **编辑** 按钮
2. 修改关联字段的选中项（增加或移除勾选）
3. 点击 **确定** 提交

系统会自动处理双向同步：
- 新增的关联：对方记录自动添加反向关联
- 移除的关联：对方记录自动移除反向关联

**删除记录时清理关联：**

删除一条记录后，该记录在所有关联方的引用会被自动清除。

### 1.5 双向同步原理

以修改「用例A」的所属模板从 `[模板1, 模板3]` 改为 `[模板1, 模板2]` 为例：

```
操作前：
  用例A.templates = [模板1, 模板3]
  模板1.cases = [用例A]
  模板2.cases = []
  模板3.cases = [用例A]

用户操作：将用例A的所属模板改为 [模板1, 模板2]

系统自动处理：
  1. 用例A.templates → [模板1, 模板2]     ← 直接更新
  2. 模板1.cases → [用例A]                 ← 不变（未移除）
  3. 模板2.cases → [用例A]                 ← 自动添加（新增关联）
  4. 模板3.cases → []                      ← 自动移除（取消关联）

操作后：
  用例A.templates = [模板1, 模板2]
  模板1.cases = [用例A]
  模板2.cases = [用例A]
  模板3.cases = []
```

### 1.6 注意事项

1. **字段名对应**：两侧配置的字段名和反向字段名必须严格匹配，否则双向同步不生效
2. **先创建页面配置**：目标集合的页面配置必须已存在，才能在下拉中选择
3. **先有数据再关联**：关联选择控件会加载目标集合的现有记录作为选项
4. **关联数据独立存储**：关联关系存储在独立的关联表中，不影响记录本身的数据字段
5. **表格显示**：关联字段在数据表格中以 `N 条关联` 的形式显示，无关联时显示 `-`

## 二、数据引用（reference）

### 2.1 特性说明

数据引用是一种**一对多单向父子关系**，核心特性：

- **单向关系**：子记录存储父记录 ID，父记录不感知子记录
- **单选**：一条子记录只能引用一个父记录
- **字段继承**：子记录可自动继承父记录的字段值显示
- **删除保护**：被引用的父记录不能直接删除，需先解除引用

### 2.2 适用场景

- 任务归属项目：一个任务属于某个项目，项目下有多个任务
- 子任务归属主任务：子任务继承主任务的部分属性
- 评论归属文章：评论继承文章标题显示
- 订单归属客户：一个客户有多条订单记录

### 2.3 配置步骤

以「任务」归属「项目」为例。

**第一步：给项目添加数据页配置**

1. 确保「项目」页面配置已存在，并有显示字段（如「项目名称」）

**第二步：给任务添加引用字段**

1. 进入 **系统配置 > 页面配置**
2. 在左侧列表选中「任务」
3. 点击 **添加字段**，填写：
   - **显示名称**：`所属项目`
   - **字段名**：`projectId`
   - **控件类型**：选择 `数据引用`
4. 底部出现**引用配置**区域，填写：
   - **目标集合**：下拉选择 `项目`
   - **显示字段**：下拉选择 `项目名称`
   - **继承字段**：勾选要继承显示的父字段（如「项目状态」「项目负责人」）
5. 点击 **确定**，再点击 **保存配置**

### 2.4 使用方式

**新增子记录时选择父记录：**

1. 进入「任务」页面，点击 **新增**
2. 在「所属项目」字段中选择一个项目（单选下拉）
3. 填写其他字段后点击 **确定** 提交

继承字段效果：
- 子记录保存后，「所属项目」显示项目名称
- 勾选的继承字段会显示父记录对应字段的值（只读）
- 继承字段在表单中以灰色背景显示，不可编辑

**表格显示：**

- 引用字段显示父记录的显示字段值（如项目名称）
- 继承字段作为额外列显示，来源于父记录
- 点击引用值可跳转到父记录详情

### 2.5 删除保护机制

当尝试删除一个被引用的父记录时：

```
场景：项目A 有任务1、任务2、任务3 三个子记录引用

操作：点击删除项目A

结果：
  - 系统检测到项目A被 3 条任务记录引用
  - 弹窗提示：「该项目被 3 条任务引用，无法删除。请先解除引用。」
  - 删除操作被阻止

解决方法：
  1. 先将任务1、任务2、任务3的「所属项目」改为其他项目或清空
  2. 确认项目A不再被任何记录引用后，方可删除
```

### 2.6 注意事项

1. **单向性**：修改子记录的引用不影响父记录，父记录不感知子记录列表
2. **单选限制**：一条记录只能引用一个父记录，不支持多选
3. **继承字段只读**：继承的字段值来源于父记录，在子记录表单中不可编辑
4. **删除需谨慎**：被引用的父记录受保护，需先清理子记录引用才能删除
5. **继承字段更新**：父记录字段值变更后，子记录的继承字段显示会自动更新（实时读取）

## 三、引用选择（quoteSelect）

### 3.1 特性说明

引用选择是一种**单向多选引用关系**，核心特性：

- **单向关系**：源记录引用目标记录，目标记录不感知
- **多选支持**：一条记录可引用多条目标记录
- **直接存储**：引用 ID 数组直接存储在记录的 JSONB 字段中
- **无删除保护**：目标记录删除后，源记录的引用 ID 保留（显示为原始 ID）

### 3.2 适用场景

- 巡检记录引用多条巡检用例：记录执行了哪些用例检查
- 任务关联多个参考文档：任务参考了哪些文档
- 订单选择多个优惠规则：订单适用哪些优惠
- 报告引用多条数据来源：报告使用了哪些数据

### 3.3 配置步骤

以「巡检记录」引用「巡检用例」为例。

1. 进入 **系统配置 > 页面配置**
2. 在左侧列表选中「巡检记录」
3. 点击 **添加字段**，填写：
   - **显示名称**：`引用用例`
   - **字段名**：`quotedCases`
   - **控件类型**：选择 `引用选择`
4. 底部出现**引用选择配置**区域，填写：
   - **目标集合**：下拉选择 `巡检用例`
   - **显示字段**：下拉选择 `用例名称`
5. 点击 **确定**，再点击 **保存配置**

### 3.4 使用方式

**新增/编辑记录：**

在表单中，引用选择字段以多选下拉框形式呈现：
- 支持搜索过滤目标记录
- 可选择多条目标记录
- 可清空选择
- 选择的顺序会被保留

**表格展示：**

- 引用记录以 Tag 标签形式展示
- 最多显示 3 个标签，超出部分显示 "+N" 计数
- 点击标签可跳转到引用记录所在页面
- 如果被引用的记录已删除，表格中会直接显示其 ID

**搜索：**

全局搜索可匹配引用记录的显示名称。

**导出/导入：**

- **导出**：引用选择列显示目标记录的主键值，用"、"分隔
- **导入**：在单元格中填写目标记录的主键值，用"、"分隔

### 3.5 顺序保证

引用选择字段**保证引用值的顺序在整个生命周期中保持不变**。

| 环节 | 实现方式 | 顺序保证 |
|------|---------|---------|
| 表单选择 | 多选 v-model 绑定数组 | 新选项追加到末尾，取消选择不改变其余项的相对顺序 |
| 保存到后端 | JSON 数组存入 JSONB | JSONB 保持数组元素顺序 |
| 读取展示 | JSONB → JSON 响应 → 数组解析 | 每一步都保持数组迭代顺序 |
| 表格 Tag | v-for 按数组顺序渲染 | 从左到右按顺序展示 |
| Excel 导出/导入 | 数组按序拼接/分割 | 按原始顺序处理 |

**删除场景的顺序保持：**

```
初始状态：A 引用 [B, C, D]（按此顺序保存）

场景1 — 编辑时取消选择 C：
  表单从 [B, C, D] 变为 [B, D]
  保存后：A 引用 [B, D]（B 和 D 的相对顺序不变）

场景2 — 目标集合中 C 被删除：
  A 的存储数据仍为 [B, C, D]
  表格展示：B（名称）、id-C（ID fallback）、D（名称）
  下次编辑并保存时：C 的 ID 自动清除为 [B, D]
```

### 3.6 注意事项

1. **单向性**：删除目标记录不会自动清理引用方的数据
2. **无删除保护**：引用选择不会阻止目标记录被删除
3. **删除后显示**：如果被引用的记录已删除，表格中会显示原始 ID
4. **直接存储**：引用数据存储在记录本身，不需要额外的后端接口或关联表
5. **顺序不变**：引用值的顺序始终保持创建时的顺序

> **删除目标后如何恢复引用**
>
> 引用选择存的是目标记录的**内部 ID**。删除目标后，源记录里的 ID 变成"悬空"（表格显示原始 ID）。
>
> - **用相同内部 ID 重新导入/新建目标 → 引用自动恢复**：文件导入时让导入文件带 `id` 列、填回目标原来的内部 ID；或通过 API（`POST /collections/{collection}`、`batch-create`）显式传相同 `id`。目标记录一旦以原 ID 重新存在，源记录里那个 ID 立即重新生效，无需任何额外操作。
> - **若重导的目标是全新 ID（默认导入会生成新 ID）→ 无法靠「重新解析引用」恢复**：源记录存的是已失效的旧内部 ID，它既不是目标的主键值、也不是显示值，匹配不回去。此时要么按上一条用原 ID 重导，要么手动把源记录该字段改填目标的主键值/显示值后再「重新解析引用」。

## 四、三种关系对比总结

### 4.1 功能对比

| 特性 | 关联关系 | 数据引用 | 引用选择 |
|------|---------|---------|---------|
| 方向 | 双向自动同步 | 单向（父→子） | 单向（仅源端） |
| 数量 | 多对多 | 一对多（单选） | 一对多（多选） |
| 字段继承 | 否 | 是 | 否 |
| 删除保护 | 否（自动清理） | 是（阻止删除） | 否（保留 ID） |
| 存储位置 | 独立关联表 | 子记录 JSONB | 源记录 JSONB |
| 目标感知 | 双方都感知 | 仅父感知子数量 | 目标不感知 |
| 顺序保证 | 否 | 单值无顺序 | 是 |

### 4.2 选择建议

| 业务需求 | 推荐类型 | 原因 |
|---------|---------|------|
| 双方都需要看到关联列表 | 关联关系 | 双向同步特性 |
| 子记录需要继承父记录属性 | 数据引用 | 字段继承功能 |
| 需要保护父记录不被误删 | 数据引用 | 删除保护机制 |
| 一对多，只需源端记录引用 | 引用选择 | 简单直接，无需关联表 |
| 多选引用，无需目标感知 | 引用选择 | 多选支持，直接存储 |
| 需要保持引用顺序 | 引用选择 | 顺序保证机制 |

### 4.3 典型配置示例

**关联关系配置（用例 ↔ 模板）：**

```json
// 用例侧
{
  "fieldName": "templates",
  "controlType": "relation",
  "relationConfig": {
    "targetCollection": "templates",
    "displayField": "templateName",
    "targetField": "cases"
  }
}

// 模板侧
{
  "fieldName": "cases",
  "controlType": "relation",
  "relationConfig": {
    "targetCollection": "cases",
    "displayField": "caseName",
    "targetField": "templates"
  }
}
```

**数据引用配置（任务 → 项目）：**

```json
{
  "fieldName": "projectId",
  "controlType": "reference",
  "referenceConfig": {
    "targetCollection": "projects",
    "displayField": "projectName",
    "inheritFields": ["projectStatus", "projectOwner"]
  }
}
```

**引用选择配置（巡检记录 → 用例）：**

```json
{
  "fieldName": "quotedCases",
  "controlType": "quoteSelect",
  "quoteConfig": {
    "targetCollection": "cases",
    "displayField": "caseName"
  }
}
```
