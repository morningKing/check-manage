# 跨页可自定义工作流引擎 使用说明

> 适用版本：`feat/neutral-slate-ui-redesign` 分支（子系统 B）
> 配套设计/实现：`docs/superpowers/specs/2026-06-14-cross-page-workflow-engine-design.md`、`docs/superpowers/plans/2026-06-14-cross-page-workflow-engine.md`

## 1. 它解决什么

把多个**数据页**串成一条命名的业务流程：不同**角色**在不同**阶段**处理，记录推进到下一阶段时**自动在下一页生成下游记录**（按字段映射携带上游数据并反向关联），支持**驳回**回退上一阶段，并按角色聚合**待办**。

模型是「记录递传链」——流程不是一个独立的"案件"对象，而是**一条跨页关联的记录链**，每页记录有自己的状态。

![工作流流程示意图](./workflow-engine-flow.svg)

## 2. 核心概念

| 概念 | 说明 |
|------|------|
| **工作流定义** `workflow_definitions` | 一个命名流程 = 有序的若干**阶段**。 |
| **阶段** stage | 绑定一个数据页（collection）+ 该页的一个**状态字段** + 一个**推进转换**（from→to）+ 可选**回退转换** + **办理角色** + **下游生成映射**（spawn）。 |
| **实例** `workflow_instances` | 一次运行中的流程。记录 `chain`（走过的每个阶段对应的真实记录）、`active_stages`（**当前并发活动分支**，v2）、`current_stage_id`（兼容用）、`history`（推进/驳回轨迹）。 |
| **流向边 / 条件 / 并行**（v2） | 画布上**拖连线**定义阶段流向（DAG）；推进边可配**单条件**（`字段 op 值`，留空=默认边）。推进时命中条件的边**全取（并行扇出到多个分支）**、否则走默认边；一条记录可同时跑多个并发分支，**所有分支结束**实例才完成。不画边则按阶段顺序当线性链（旧定义不变）。 |
| **推进 / 驳回** | 推进 = 当前阶段记录的状态字段做"推进转换" → 按出边路由生成下游记录、对应分支前移、通知下阶段角色；驳回 = 做"回退转换" → 该分支退回 reject 边目标、复位上游记录状态、通知上阶段角色。 |
| **待办收件箱** | 列出"当前阶段办理角色 ∋ 我"的运行中实例及其当前记录。 |

## 3. 前置条件（重要）

推进/驳回按钮**复用现有的字段状态机**（`workflowConfig`）。所以在配置工作流之前，先在**页面配置**里为每个阶段要用的**状态字段**设好 `workflowConfig.transitions`（即合法的状态转换 + 角色 + 条件）。工作流设计器只**引用**这些转换，不代写。

- 状态字段须为 `select` 类型且 `workflowConfig.enabled = true`。
- 工作流阶段的"推进转换 from→to"必须是该字段 `workflowConfig` 里存在的一条合法转换。

## 4. 配置一个工作流（设计器）

入口：**设置中心 → 结构配置 → 工作流**（需 `admin.workflows` 权限）。点「新建工作流」：

1. **名称 / 描述 / 启用**。
2. **阶段**（有序，可上移/下移/删除/添加）。每个阶段填：
   - **阶段名**：如"需求评审"。
   - **绑定数据页**：下拉选一个数据页（collection）。
   - **状态字段**：选该页带 `workflowConfig` 的状态字段（下拉，亦可手填）。
   - **推进转换** from → to：如 `待评审 → 已通过`。
   - **回退转换** from → to（可选）：如 `设计中 → 退回`；留空则该阶段不可驳回。
   - **办理角色**：多选，本阶段谁能办理（按角色）。
   - **生成下游**（spawn）：键值映射，决定推进时在**下一阶段页**新建记录的字段：
     - 目标字段 ← 取值表达式（见 §5）。
     - linkBack 字段：下游记录上用于反指上游记录 id 的字段（建立递传链关联）。
   - 末阶段无需 spawn（推进即"流程完成"）。
3. 保存。

## 5. 字段映射取值语法（spawn）

| 表达式 | 含义 |
|--------|------|
| `$source.<字段名>` | 上游记录该字段的值 |
| `$source.id` | 上游记录的 id（常用于 linkBack 反向关联） |
| `$operator` | 当前操作人用户名 |
| `$NOW` | 当前时间（ISO8601） |
| 其它字面量 | 原样写入 |

示例（需求 → 设计）：`title ← $source.title`，`srcReq ← $source.id`。

## 6. 运行流程

1. **启动**：在首阶段数据页打开一条记录的「查看」详情，点「启动工作流」→ 选择以本页为首阶段的工作流 → 创建实例，通知首阶段角色。
2. **办理 / 推进**：办理人在记录详情上看到「推进到 X」按钮（按角色/条件过滤）。点击 → 弹「意见」对话框（可选填）→ 确认。
   - 系统校验状态机合法性 → 应用状态转换 → 工作流引擎在**同一事务**内识别"这是阶段推进" → 按映射在下一页生成下游记录 → 实例前移 → 通知下一阶段角色。
   - 下游记录的自增编号由后端**原子分配**（与并发一致性加固对齐，不会重号）。
3. **待办**：顶栏「我的待办」图标 → 列出归我办理的当前阶段记录 → 「去处理」跳转到对应数据页并定位高亮该记录。
4. **驳回**：在驳回类转换按钮上点击 → 填意见 → 实例退回上一阶段、上游记录状态复位为可重办、通知上阶段角色。被驳回后上游重新推进可继续前进（实例按当前阶段定位，不会卡死）。
5. **完成**：末阶段推进 → 实例 `status = completed`。

## 7. 权限（RBAC）

- **管理工作流定义**：`admin.workflows`（设计器 + 定义 CRUD）。
- **办理阶段**：阶段 `assignedRoles` + 该状态字段转换自身的 `roles` 双重门禁。
- **收件箱**：按当前用户角色过滤。

## 8. 数据模型与一致性

- `workflow_definitions(id, name, description, enabled, stages jsonb)` —— `stages` 为有序数组（见 §2）。
- `workflow_instances(id, workflow_id, status, current_stage_id, chain jsonb, history jsonb, started_by, ...)`。
  - `chain`：`[{stageId, collection, recordId, enteredAt, completedBy}]` —— 跨页递传链，用于追溯与驳回定位。
  - 实例定位按 `current_stage_id` 对应的 chain 项匹配（保证驳回后再推进仍能找到实例）。
- **并发**：`on_transition` 对实例行 `SELECT … FOR UPDATE`，同一实例的并发推进/回退被串行化，避免重复生成下游。
- **下游记录生成**：复用 `allocate_sequence`（原子序号）+ 主键 advisory lock，**与单建/导入/合并/还原一致**，不会与后续创建重号。
- **不污染业务数据**：推进/驳回意见 `_workflowComment` 仅用于驱动引擎，不写入记录数据；驳回意见以 `_rejectComment` 写回被驳回回去的上游记录（便于办理人看到原因）。

## 9. 部署 / 迁移

新增一张定义表 + 一张实例表。对**现有数据库**执行幂等迁移（见 `../integration/data-migration.md` 同款流程，**先备份**）：

```bash
cd server
python -m migrations.2026_06_14_workflow_tables   # 预期输出 {'status': 'ok'}
```

全新安装经 `python init_db.py` 已含这两张表，无需单独迁移。

## 10. 范围与进度

**MVP（已完成）**：线性多阶段跨页流程 + 记录递传链 + 推进 + 简单驳回 + 按角色指派 + 待办收件箱 + 设计器。

**v2 — 分支/并行（已完成）**：图形化流程设计器（画布拖拽建节点/连线）+ DAG + 条件边引擎 + 并行多路分发（一个阶段扇出到多个并发分支，`active_stages` 追踪活动分支）。

**仍待开发（后续）**：

## 11. 测试覆盖

后端：阶段推进 + 下游生成、驳回 + 上游复位、驳回后再推进、末阶段完成、角色门禁、`update_item` 钩子端到端（真实 PUT）、启动实例 API、定义 CRUD、收件箱按角色过滤、`_workflowComment` 不入库。前端：设计器 store、收件箱渲染。端到端：Playwright 实测设计器加载/编辑真实工作流 + 收件箱显示真实待办。

已知后续可补：同一实例并发双推进竞态测试、驳回→再推进的整路 PUT 端到端、前端意见对话框组件测试。
