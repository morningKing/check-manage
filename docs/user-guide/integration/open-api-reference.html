Open API Reference

让外部系统安全地读写你的业务数据

通过 API Key 认证的 RESTful 接口，外部系统可以查询、新增、修改数据集合中已授权开放的记录 —— 包括附件文件的上传与下载。所有接口收发 application/json。

Base URLhttp://<host>:<port>/api/v1
开发环境http://localhost:7001/api/v1
认证头X-API-Key: cm_xxx
数据格式application/json

默认情况下集合仅对 Open API 只读开放，写入需管理员额外开启「允许写入」。文件上传走独立的 multipart/form-data 端点，详见下方「文件上传 & 文件字段录入」。

Authentication

认证

所有请求都必须在请求头中携带有效的 API Key。密钥由系统哈希后比对，不会以明文存储。

headers
X-API-Key: cm_A1b2C3d4E5f6G7h8I9j0K1l2M3n4O5p6Q7r8S9t0U1v2

密钥格式

属性说明
前缀cm_
长度约 46 个字符
字符集URL 安全的 Base64 字符

认证过程

  • 系统对密钥做 SHA-256 哈希

    请求头中的 cm_xxx 不会明文比对，而是先做哈希再查库。

  • 查 api_keys 表比对哈希

    命中则继续，未命中直接返回 401。

  • 检查密钥状态

    若密钥已被管理员停用，返回 401「API key has been revoked」。

  • 更新最近使用时间并放行

    写入 last_used_at，供管理员追踪密钥使用情况。

认证错误

状态码error触发条件
401Missing API key请求头未携带 X-API-Key
401Invalid API keyAPI Key 无效（不存在）
401API key has been revokedAPI Key 已被管理员停用

Before you start

前置条件

调用 Open API 前，管理员需要先在系统内完成三件事。只有满足条件的集合才会出现在 GET /collections 的返回结果中。

  • 创建 API Key

    「系统配置 → 数据工具 → Open API」中创建，创建后仅显示一次密钥明文。

  • 开放数据集合

    在「页面配置」中勾选「允许 Open API 访问」（api_public = true）。

  • 开放写入权限（可选）

    如需新增 / 修改数据，额外开启「允许写入」（api_writable = true）。

Get connected

接入流程

从创建密钥到外部系统跑通第一个请求，共 7 步。

  • 管理员创建 API Key

    进入「系统配置 → 数据工具 → Open API」，点击「创建 API Key」并命名。

  • 立即复制并保存密钥

    密钥仅在创建时显示一次，关闭弹窗后将无法再次查看明文。

  • 开启目标页面的「允许 Open API 访问」

    进入「系统配置 → 平台管理 → 页面配置」，选中需要开放的数据页并保存。

  • 需要写入时，额外开启「允许写入」

    默认仅只读；外部系统需要新增 / 修改数据时才开启。

  • 把密钥和 Base URL 交给外部系统开发者

    通过安全渠道传递，不要通过明文邮件或聊天记录长期留存。

  • 外部系统调用 GET /collections 验证连通性

    返回 200 说明密钥有效、网络可达。

  • 按需调用数据 / 文件接口

    连通性确认后即可正式接入业务查询与写入逻辑。

5.6

新增记录

POST /api/v1/collections/{collection} 需允许写入

不传 id 则自动生成 api-xxxxxxxxxxxx 格式的记录 ID。

request body
{
  "caseName": "新增用例",
  "priority": "high",
  "status": "active"
}
response · 201
{
  "data": { "id": "api-a1b2c3d4e5f6", "caseName": "新增用例", "branchId": "main" }
}

引用类字段（reference / quoteSelect）可以延迟解析。推荐直接写目标记录的内部 ID；如果被引用的数据还没创建，也可以先写目标的主键值 / 显示字段值作为原始值（quoteSelect 例如 ["苹果手机", "华为笔记本"]），系统会原样保存不报错。

待数据补齐后，到该数据页点击「更多 → 重新解析引用」即可把原始值补全为内部 ID。relation（多对多）不支持此机制，需在请求体中用 _relations 按内部 ID 写入。

状态码error触发条件
400Validation failed必填字段缺失（details 数组列出具体字段）
403Collection is read-only集合未开启「允许写入」
409Record ID already exists自定义 ID 已存在（同一分支下）
409Primary key conflict主键字段值与已有记录重复

5.7

批量新增记录

POST /api/v1/collections/{collection}/batch 需允许写入

一次请求创建多条记录（单次最多 1000 条）。create-only，不做 upsert——批次里任何一条 id 已存在都算该条失败，不会更新已有记录。

request body
{
  "records": [
    { "id": "prod-001", "name": "产品A" },
    { "name": "产品B" }
  ],
  "options": { "continueOnError": true }
}
response · 201
{
  "data": [ { "id": "prod-001", ... } ],
  "created": 2,
  "failed": 0
}

options.continueOnError 默认 false：任何一条记录失败就整批 400、不写入；传 true 则跳过失败记录，成功的照常写入，响应体 errors 数组列出失败详情。

状态码error触发条件
400records is requiredrecords 为空数组或缺失
400Batch size exceeds maximum of 1000 records单次请求记录数超过 1000
400Validation failed for one or more recordscontinueOnError=false 且至少一条记录校验失败
403Collection is read-only集合未开启「允许写入」

单条记录失败时，errors[].error 可能是以下两种不同的 ID 冲突之一，不要混淆：

errors[].error触发条件
Record ID already exists该记录的 id 已存在于数据库中——id 在同一分支下跨集合全局唯一，即便冲突的记录属于另一个集合也会命中此错误
Duplicate ID in batch该 id 是本次请求 records 数组内部重复出现的值，与数据库已有数据无关

5.8

修改记录

PUT /api/v1/collections/{collection}/{id} 需允许写入

支持部分更新：只传需要修改的字段，未传字段保持原值。

request body
{ "status": "inactive", "_version": 1 }

传入 _version 时进行乐观锁并发冲突检测。

状态码error触发条件
409Record has been modified...乐观锁冲突（code: "VERSION_CONFLICT"）
404Record not found记录 ID 不存在，或在该分支下不存在

5.9 · 深入讲解

文件上传 & 文件字段录入

file / image 类型的字段在数据库里存的不是二进制内容，而是一个文件对象数组。 所以往这类字段写数据分两步：先把文件本体传到独立的文件端点换回一个 uid， 再把包含这个 uid 的对象数组写进新增 / 修改记录的请求体里。

STEP 1

上传文件

POST /api/v1/files，multipart 表单，拿到 uid。

STEP 2

写入字段

把 uid 组装成文件对象，塞进 file/image 字段的数组。

STEP 3

下载 / 展示

记录里的 apiUrl 即为下载地址，带同一个 API Key 即可取回文件。

Step 1 · 上传文件本体

POST /api/v1/files 需允许写入

请求体为 multipart/form-data，而非 JSON —— 这是 Open API 里唯一一个不收 JSON 请求体的接口。

表单字段必填说明
file是要上传的文件本体
collection是目标数据页标识，须为「公开且允许写入」的集合
curl
# 上传本地文件 report.pdf，目标集合为 devices
curl -s -X POST \
  -H "X-API-Key: $API_KEY" \
  -F "file=@./report.pdf" \
  -F "collection=devices" \
  "$BASE_URL/files"
response · 200
{
  "data": {
    "uid": "8f3c...e21",
    "name": "report.pdf",
    "size": 20480,
    "mimeType": "application/pdf",
    "downloadUrl": "/api/v1/files/8f3c...e21/download"
  }
}
状态码error触发条件
400collection is required表单缺少 collection 字段
404Collection not found or not public目标集合非公开
403Collection is read-only目标集合未开启「允许写入」

Step 2 · 把 uid 写进 file / image 字段

拿到 uid 后，在新增 / 修改记录的请求体里，把如下对象数组写进对应字段（字段名以 schema 中定义的为准，图中用「附件」举例）：

PUT /collections/devices/{id}
{
  "名称": "外部记录1",
  "附件": [
    {
      "uid": "8f3c...e21",
      "name": "report.pdf",
      "size": 20480,
      "type": "application/pdf",
      "url": "/api/data-files/8f3c...e21/download"
    }
  ]
}

文件对象字段说明

字段类型说明
uidstring上传接口返回的文件标识，必填
namestring文件名，用于列表展示
sizeinteger字节数
typestringMIME 类型
urlstring内部下载地址，走网页端 JWT 鉴权，外部系统不可用
apiUrlstring查询记录时系统自动附带，Open API 下载专用（见下方 Step 3）

image 字段和 file 字段走同一套流程 —— 都是先 POST /files 拿 uid，再写入同样结构的对象数组，唯一区别是页面配置里各自的数量 / 大小上限不同（图片默认最多 9 张 / 单张 5MB，文件默认最多 5 个 / 单个 10MB，具体以 DATA_FILE_MAX_MB 配置为准）。

一个字段可以写入多个文件对象：多次调用 POST /files 拿到多个 uid，再把它们一起放进数组即可。

常见错误：直接把本地文件路径或 base64 写进 JSON 字段。这类字段只接受「已上传文件的引用对象数组」，不会帮你解析路径或反编码 base64 —— 必须先完成 Step 1 拿到 uid。

端到端示例：上传并挂到新记录上

以下示例先上传文件，再用拿到的 uid 新建一条记录，一步到位。

bash
# 1. 上传文件，取出 uid（依赖 jq）
UPLOAD=$(curl -s -X POST \
  -H "X-API-Key: $API_KEY" \
  -F "file=@./report.pdf" \
  -F "collection=devices" \
  "$BASE_URL/files")
FILE_UID=$(echo "$UPLOAD" | jq -r '.data.uid')
FILE_NAME=$(echo "$UPLOAD" | jq -r '.data.name')
FILE_SIZE=$(echo "$UPLOAD" | jq -r '.data.size')

# 2. 用 uid 组装文件对象，新建记录
curl -s -X POST \
  -H "X-API-Key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d "{\"名称\":\"外部记录1\",\"附件\":[{\"uid\":\"$FILE_UID\",\"name\":\"$FILE_NAME\",\"size\":$FILE_SIZE,\"type\":\"application/pdf\"}]}" \
  "$BASE_URL/collections/devices"
python
import requests

API_KEY = "cm_A1b2C3d4E5f6G7h8I9j0K1l2M3n4O5p6Q7r8S9t0U1v2"
BASE_URL = "http://localhost:7001/api/v1"
HEADERS = {"X-API-Key": API_KEY}


def upload_file(collection, file_path):
    """上传单个文件，返回可写入 file/image 字段的对象"""
    with open(file_path, "rb") as f:
        resp = requests.post(
            f"{BASE_URL}/files",
            headers=HEADERS,
            files={"file": f},
            data={"collection": collection},
        )
    resp.raise_for_status()
    meta = resp.json()["data"]
    return {
        "uid": meta["uid"],
        "name": meta["name"],
        "size": meta["size"],
        "type": meta["mimeType"],
    }


def create_record_with_attachment(collection, fields, file_field_name, file_path):
    """上传文件 + 新增一条带附件的记录"""
    file_obj = upload_file(collection, file_path)
    payload = {**fields, file_field_name: [file_obj]}
    resp = requests.post(
        f"{BASE_URL}/collections/{collection}",
        headers={**HEADERS, "Content-Type": "application/json"},
        json=payload,
    )
    resp.raise_for_status()
    return resp.json()["data"]


# 使用示例
record = create_record_with_attachment(
    "devices",
    {"名称": "外部记录1"},
    "附件",
    "./report.pdf",
)
print(f"新增成功: {record['id']}")

鉴权与限制同写记录一致：只能向公开且允许写入的集合上传（否则 404 / 403）。支持中文文件名；单文件大小上限同网页端（DATA_FILE_MAX_MB）。文件被某条记录引用后，才能经 GET /files/{uid}/download 下载。

5.10

下载文件

查询记录时，每个文件对象都会附带一个 apiUrl —— 这才是 Open API 应该用来下载的地址（区别于内部 url，后者走网页端 JWT，外部系统调用不了）。

GET /collections/devices/{id} response 片段
{
  "attachment": [
    {
      "uid": "8f3c...e21", "name": "report.pdf",
      "url": "/api/data-files/8f3c...e21/download",     // 内部地址，不可用
      "apiUrl": "/api/v1/files/8f3c...e21/download"  // ← Open API 用这个
    }
  ]
}
GET /api/v1/files/{fileId}/download

返回文件二进制流，响应头 Content-Disposition: attachment。

GET /api/v1/files/{fileId}

仅获取文件元数据（不下载内容），用于展示文件名 / 大小 / 类型。

response · 200
{
  "data": {
    "id": "8f3c...e21", "name": "report.pdf",
    "mimeType": "application/pdf", "size": 20480,
    "downloadUrl": "/api/v1/files/8f3c...e21/download"
  }
}

安全边界：只有被已开放（api_public）集合的记录引用的文件才能经 Open API 下载；挂在未开放集合上的文件返回 404 File not found。

Security

安全注意事项

密钥保管API Key 创建后仅显示一次，请妥善保存
传输安全生产环境务必使用 HTTPS，防止密钥泄露
最小开放仅对必要的集合开启 Open API，遵循最小权限原则

及时停用不再使用的密钥应立即停用或删除
速率监控系统记录每个密钥的最后使用时间，管理员可据此监控使用情况

FAQ

常见问题

文件字段能不能直接传文件路径或 base64？
不能。file/image 字段只接受「已上传文件的引用对象数组」（含 uid），必须先调用 POST /files 完成上传，再把返回的 uid 组装进对象数组写入记录。
一个文件字段能挂多个文件吗？
可以。多次调用 POST /files 拿到多个 uid，把它们各自组装成文件对象后一起放进同一个数组即可。具体数量上限以该字段在页面配置中的设置为准。
下载文件用 url 还是 apiUrl？
Open API 场景一律用 apiUrl。url 是内部地址，走网页端 JWT 鉴权，外部系统直接调用会 401。
如何判断哪些集合已开放 API 访问？
调用 GET /collections 即可，未开放的集合不会出现在返回结果中；writable 字段标识该集合是否允许写入。
能否写入数据到特定分支？
可以，在 POST/PUT 请求中加 branchId 参数即可。分支需存在且处于活跃状态，否则返回 404 Branch not found or not active。
修改记录需要传所有字段吗？
不需要。PUT 支持部分更新，只传要修改的字段，未传字段保持原值。
如何避免并发修改冲突？
PUT 请求体传入 _version 字段即可触发乐观锁检测；版本不一致会返回 409 VERSION_CONFLICT。建议先 GET 拿最新版本号，再连同 _version 一起提交修改。
引用字段指向的数据还没创建怎么办？
可以先写入目标的主键值或显示字段值作为原始值，系统会原样保存；待数据补齐后，到该数据页点击「更多 → 重新解析引用」即可补全为内部 ID。仅适用于 reference/quoteSelect，不适用于 relation。
API Key 泄露了怎么办？
立即登录系统，在「Open API」管理页面停用该密钥，然后创建一个新密钥重新分发给外部系统。
